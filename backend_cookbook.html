VerbPurposeIdempotent?Has body?FastAPI decorator
GET Fetch resource(s) Yes No @app.get
POST Create new resource No Yes @app.post
PUT Replace full resource Yes Yes @app.put
PATCH Partial update Sometimes Yes @app.patch
DELETE Remove resource Yes Rarely @app.delete
Idempotent = calling it N times = same as calling it once. Use PUT/DELETE safely for retries without side effects.

Response shape — always consistent
# Universal response wrapper (FastAPI + Pydantic) from pydantic import BaseModel from typing import Generic, TypeVar, Optional T = TypeVar("T") class APIResponse(BaseModel, Generic[T]): success: bool data: Optional[T] = None error: Optional[str] = None meta: Optional[dict] = None # pagination, timestamps # ✅ Success # { "success": true, "data": {...}, "meta": {"page":1,"total":50} } # ❌ Error # { "success": false, "error": "Post not found" } # Usage in endpoint: @app.get("/posts/{id}", response_model=APIResponse[PostSchema]) async def get_post(id: int): post = await fetch_post(id) return APIResponse(success=True, data=post)
Naming rules — do & don't
✓
Use plural nouns: /posts, /users, /comments
Resources are collections. Even for a single item endpoint: GET /posts/{id}
✓
Version in the URL: /api/v1/posts
Most visible, recommended for public APIs. Allows non-breaking changes.
✓
Filters & pagination in query params: ?page=2&tag=redis
✗
Never use verbs in URLs: /api/getPosts, /api/deleteUser
The HTTP verb IS the action. URL = the resource.
✗
Don't nest more than 2 levels: /users/1/posts/5/comments/2/likes
Instead: GET /comments/2/likes or GET /likes?comment_id=2

Auth flows — step by step
FastAPI OAuth2 + JWT — full pattern
from fastapi import Depends, HTTPException from fastapi.security import OAuth2PasswordBearer from datetime import datetime, timedelta import jwt oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token") SECRET = "your-256-bit-secret" ALGO = "HS256" def create_access_token(user_id: str) -> str: payload = { "sub": user_id, "exp": datetime.utcnow() + timedelta(minutes=15) } return jwt.encode(payload, SECRET, algorithm=ALGO) async def get_current_user(token: str = Depends(oauth2_scheme)): try: payload = jwt.decode(token, SECRET, algorithms=[ALGO]) user_id: str = payload.get("sub") if not user_id: raise ValueError("No sub claim") except jwt.ExpiredSignatureError: raise HTTPException(401, "Token expired") except jwt.InvalidTokenError: raise HTTPException(401, "Invalid token") return await db_get_user(user_id) # Protect any route: @app.get("/profile") async def profile(user = Depends(get_current_user)): return APIResponse(success=True, data=user)
Access vs refresh token strategy
15 min
Access token TTL
short-lived, in-memory
7 days
Refresh token TTL
HttpOnly cookie
✓
Store refresh token in HttpOnly cookie — XSS can't steal it
✓
Keep access token in memory only — never localStorage
✓
Store refresh token hash in Redis — enables instant revocation
✗
Never store sensitive data in JWT payload — it's base64 encoded, not encrypted
✗
Never use alg: none — always specify algorithm explicitly in decode call
Common auth mistakes avoid
✗
Returning 200 with {"error":"unauthorized"}
Always 401 for missing/invalid auth, 403 for insufficient permissions
✗
Logging full JWT tokens
Tokens are credentials. Log user_id only, never the token string.
✗
Not validating token expiry on every request
Always decode and check exp. Don't cache decoded tokens.
✗
Same secret for access & refresh tokens
Use different secrets. Limits blast radius if one is compromised.

Cache-aside pattern — FastAPI + redis-py
import redis.asyncio as redis import json r = redis.from_url("redis://localhost:6379", decode_responses=True) async def get_post(post_id: int): cache_key = f"post:{post_id}" # 1️⃣ Try cache first cached = await r.get(cache_key) if cached: return json.loads(cached) # ⚡ Cache HIT — ~1ms # 2️⃣ Miss — hit the database post = await db.fetch_post(post_id) # ~100ms if not post: raise HTTPException(404, "Post not found") # 3️⃣ Populate cache with TTL await r.setex(cache_key, 3600, json.dumps(post)) return post # Always invalidate on mutation! async def update_post(post_id: int, data: PostUpdate): await db.update_post(post_id, data) await r.delete(f"post:{post_id}") # ← crucial!

Rate limiting — fixed window with Redis
import time from fastapi import Request, HTTPException async def rate_limit( user_id: str, max_requests: int = 100, window_seconds: int = 60 ): window = int(time.time() // window_seconds) key = f"rate:{user_id}:{window}" current = await r.incr(key) if current == 1: await r.expire(key, window_seconds) # set TTL on first hit if current > max_requests: raise HTTPException( status_code=429, detail="Too many requests", headers={"Retry-After": str(window_seconds)} ) # Use as a FastAPI dependency: @app.get("/feed") async def get_feed(user = Depends(get_current_user)): await rate_limit(user.id, max_requests=60) ...

Lifespan — startup & shutdown (modern pattern)
from contextlib import asynccontextmanager from fastapi import FastAPI import redis.asyncio as aioredis @asynccontextmanager async def lifespan(app: FastAPI): # ── startup ────────────────────────────────── app.state.redis = await aioredis.from_url( settings.REDIS_URL, max_connections=20, decode_responses=True ) await database.connect() yield # ← app is running here # ── shutdown ───────────────────────────────── await app.state.redis.close() await database.disconnect() app = FastAPI(lifespan=lifespan, title="My API") # Access Redis anywhere via dependency: def get_redis(request: Request): return request.app.state.redis

Production checklist before deploy
✓
All secrets in env vars via pydantic BaseSettings — never hardcoded
✓
Redis connection pool with max_connections — not new connection per request
✓
HTTPS only — secure=True + httponly=True on all cookies
✓
Rate limiting on /auth/token — prevent brute force (5 req/min)
✓
Structured logging — include request_id, user_id, duration_ms, status
✓
Health check: GET /health pings DB + Redis, returns latency
✓
Graceful shutdown via lifespan — closes all connections cleanly
✓
Cache invalidation on every write/delete — stale cache = subtle bugs
✗
Don't expose stack traces in 500 responses — log internally, return generic message
✗
Don't ignore 422 Unprocessable Entity — add request logging middleware to catch it
