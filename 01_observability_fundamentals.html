Observability fundamentals

Before writing a single line of observability code, you need the mental model. Logs, traces, and metrics are not interchangeable — each answers a different question. Confuse them and you'll instrument the wrong thing for the wrong reason.

Logs — the event record

A log is a timestamped record of a discrete event. Think of it as your system's diary. Every error, every user action, every unexpected condition produces a log line. The critical shift from beginner to senior is moving from plain text logs to structured JSON logs — because structured logs are queryable at scale.

# Hard to filter, parse, or alert on at scale 2024-01-15 14:32:01 ERROR Failed to process request for user 42 2024-01-15 14:32:02 INFO Request completed in 4821ms
# Each field is independently queryable { "timestamp": "2024-01-15T14:32:01.234Z", "level": "ERROR", "logger": "rag_service", "message": "LLM call failed", "user_id": "u_abc123", "workspace_id":"ws_xyz", "latency_ms": 4821, "error": "timeout after 5000ms" }
✓ Use logs for
Debugging a specific error or exception
Auditing what a specific user did
Recording rare but important events
Capturing stack traces on failure
✗ Don't use logs for
High-frequency per-request stats
Aggregated request counts (→ metrics)
Request flow breakdown (→ traces)
Alerting on thresholds (→ metrics)

Traces — the request journey

A trace follows a single request as it moves through your system. Each logical step is a span. Spans form a parent-child tree — a span can contain child spans, each measuring a sub-step. Together they show you exactly where time was spent inside one specific request.

# One trace = one request end-to-end Trace: rag.answer (trace_id: a1b2c3d4) ├─ span: retrieval [ 0ms → 210ms] chunks_retrieved=8 ├─ span: context_building [210ms → 245ms] chunks_used=5 └─ span: llm_generation [245ms → 2100ms] tokens=842, cost=$0.004 ↑ this is where the slowness is!
✓ Use traces for
Pinpointing which step in a request is slow
Tracking LLM token usage and cost per call
Debugging AI pipeline quality issues
Understanding multi-step async flows
✗ Don't use traces for
System-wide health at a glance (→ metrics)
Trend analysis over days (→ metrics)
Alerting on error rates (→ metrics)
Reading error messages (→ logs)

Metrics — the aggregate numbers

Metrics are numeric measurements collected over time. Unlike logs (one event = one record), metrics are aggregated — "500 requests in the last minute", not each request individually. They're cheap to store, fast to query, and the backbone of dashboards and alerting.

# COUNTER — only ever increases. Use for totals. http_requests_total{status="200"} → 45821 http_requests_total{status="500"} → 12 # GAUGE — goes up and down. Use for current state. active_connections → 47 memory_usage_bytes → 142000000 # HISTOGRAM — measures distributions. Use for latency. request_duration_seconds_bucket{le="0.5"} → 3821 request_duration_seconds_bucket{le="1.0"} → 4512 request_duration_seconds_bucket{le="+Inf"}→ 4521
# Rate of requests per second over last 5 minutes rate(http_requests_total[5m]) # Error rate (5xx responses only) rate(http_requests_total{status=~"5.."}[5m]) # P95 latency across all requests histogram_quantile(0.95, rate(request_duration_seconds_bucket[5m]) ) # P99 latency for chat endpoint only histogram_quantile(0.99, rate(request_duration_seconds_bucket{handler="/ai/chat"}[5m]) )
✓ Use metrics for
Trending over time (hours, days, weeks)
Alerting on thresholds (P99 > 2s, error rate > 1%)
Capacity planning and resource usage
SLA / uptime reporting
✗ Don't use metrics for
Debugging a specific failed request
Seeing what a specific user did
Understanding per-call latency breakdown

The right mental model

Metrics tell you something is wrong. Traces tell you where in your code the problem is. Logs tell you the exact error message and context.

You always start with metrics (alert fires), drill into traces (find the slow span), then read logs (understand the actual error). Each layer answers a more specific question than the last.

Real debugging scenario — "AI chat is slow"

1
Metric Alert fires: P95 latency > 3s
Grafana dashboard shows a spike starting at 14:30. You know something is wrong and roughly when. But not why or where. This is all metrics can tell you.
2
Trace Open Langfuse, filter traces in that 14:30–14:45 window
You see that the llm_generation span averages 4.2s. The retrieval span is normal at 180ms. Now you know where: the bottleneck is inside the LLM call, not your vector search.
3
Log Search structured logs for slow requests in that window
Filter by latency_ms > 3000 and logger = "rag_service". You find a warning: "token budget exceeded, truncating to 8192 tokens". A recent note update added very large content, bloating context. Now you know the exact cause.

Why structured logging changes everything

Plain text logs require you to grep and parse manually. At 100 requests/second, that's 6,000 log lines per minute — you cannot read those. Structured JSON logs let log aggregation tools (Datadog, Loki, CloudWatch) index every field, so you can ask: "show me all ERROR logs for workspace_id=ws_xyz in the last 10 minutes with latency_ms > 2000" — instantly.

# Python: how your observability/logging.py works import logging, json, datetime class JSONFormatter(logging.Formatter): def format(self, record): log = { "timestamp": datetime.datetime.utcnow().isoformat(), "level": record.levelname, "logger": record.name, "message": record.getMessage(), } # merge in any extra fields passed by the caller for key in ("request_id", "workspace_id", "user_id", "latency_ms"): if hasattr(record, key): log[key] = getattr(record, key) return json.dumps(log) # Usage anywhere in the codebase: logger = get_logger("rag_service") logger.warning("LLM slow", extra={"latency_ms": 4821, "user_id": uid})
