Core DB Design Concepts

Key Terms

  • ▸
    Table — a collection of related data in rows & columns (also called a relation)
  • ▸
    Row / Record — a single data entry
  • ▸
    Column / Field — an attribute of data
  • ▸
    Schema — the blueprint/structure of the DB
  • ▸
    Query — a request to retrieve or modify data

Keys

  • ▸
    Primary Key (PK) — uniquely identifies each row; never NULL
  • ▸
    Foreign Key (FK) — references a PK in another table; enforces referential integrity
  • ▸
    Composite Key — PK made of 2+ columns
  • ▸
    Unique Key — like PK but can be NULL
  • ▸
    Surrogate Key — artificial PK (e.g. auto-increment ID)

Constraints

  • ▸
    PRIMARY KEY — unique + not null
  • ▸
    FOREIGN KEY — referential integrity
  • ▸
    UNIQUE — no duplicates in column
  • ▸
    NOT NULL — field is required
  • ▸
    CHECK — custom rule validation
  • ▸
    DEFAULT — fallback value
02

Normalization (Normal Forms)

1NF → 3NF Rules

  • 1NF
    Each column holds atomic (single) values. No repeating groups. Table has a PK.
  • 2NF
    Must be 1NF. Every non-key column fully depends on the entire PK (eliminates partial dependency).
  • 3NF
    Must be 2NF. No transitive dependencies — non-key columns depend only on the PK, not on other non-key columns.
  • BCNF
    Stricter 3NF. Every determinant must be a candidate key.

Why Normalize?

Normalization reduces data redundancy, prevents update/insert/delete anomalies, and keeps data consistent across the database.

BAD — redundant data:

-- OrderID | Customer | CustCity | Product
1        | Alice    | Dhaka    | Laptop
2        | Alice    | Dhaka    | Mouse  ← duplicated

GOOD — normalized:

-- Customers: CustomerID, Name, City
-- Orders:    OrderID, CustomerID(FK), Product
03

Relationships & Cardinality

One-to-Many (1:M)

Most common. One record in table A links to many in table B.

-- One Customer → Many Orders
Customers (CustomerID PK)
Orders    (OrderID PK, CustomerID FK)

Many-to-Many (M:N)

Requires a junction/bridge table with two FKs.

-- Students ↔ Courses
Enrollments (
  StudentID FK,
  CourseID  FK,
  PRIMARY KEY(StudentID, CourseID)
)

One-to-One (1:1)

Each row in A matches exactly one row in B. Used to split large tables or store optional data.

-- User ↔ UserProfile
UserProfiles (
  UserID PK FK REFERENCES Users
)

DDL — Data Definition Language

CREATE TABLE

CREATE TABLE users (
  id         INT          PRIMARY KEY AUTO_INCREMENT,
  username   VARCHAR(50)  NOT NULL UNIQUE,
  email      VARCHAR(100) NOT NULL,
  age        INT          CHECK (age >= 0),
  status     VARCHAR(20)  DEFAULT 'active',
  created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);

ALTER & DROP

-- Add column
ALTER TABLE users
  ADD COLUMN phone VARCHAR(15);

-- Modify column
ALTER TABLE users
  MODIFY COLUMN email VARCHAR(200);

-- Drop column
ALTER TABLE users
  DROP COLUMN phone;

-- Rename table
ALTER TABLE users
  RENAME TO accounts;

-- Delete table entirely
DROP TABLE users;

-- Remove all rows, keep structure
TRUNCATE TABLE users;

Foreign Key Example

CREATE TABLE orders (
  order_id    INT           PRIMARY KEY AUTO_INCREMENT,
  customer_id INT           NOT NULL,
  total       DECIMAL(10,2) NOT NULL,
  ordered_at  TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (customer_id) REFERENCES users(id)
    ON DELETE CASCADE    -- delete orders when user is deleted
    ON UPDATE CASCADE    -- update FK when PK changes
);
ON DELETE CASCADE ON DELETE SET NULL ON DELETE RESTRICT ON UPDATE CASCADE

DML — Data Manipulation Language

INSERT

-- Single row
INSERT INTO users (username, email, age)
VALUES ('alice', 'alice@mail.com', 28);

-- Multiple rows
INSERT INTO users (username, email)
VALUES
  ('bob',   'bob@mail.com'),
  ('carol', 'carol@mail.com');

-- Insert from SELECT
INSERT INTO archive_users
SELECT * FROM users
WHERE status = 'inactive';

UPDATE & DELETE

-- Update specific rows
UPDATE users
SET   status = 'inactive',
      age    = age + 1
WHERE id = 5;

-- ⚠️  Always use WHERE with UPDATE
-- Without it, ALL rows are updated!

-- Delete specific rows
DELETE FROM users
WHERE status = 'inactive'
  AND created_at < '2020-01-01';

SELECT — Querying Data

SELECT Anatomy

SELECT   col1, col2, COUNT(*)  -- 1. what to return
FROM     table_name             -- 2. where from
JOIN     other_table ON ...     -- 3. joins
WHERE    condition               -- 4. filter rows
GROUP BY col1                   -- 5. aggregate
HAVING   COUNT(*) > 1          -- 6. filter groups
ORDER BY col2 DESC             -- 7. sort
LIMIT    10 OFFSET 20;         -- 8. paginate
Logical execution order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

WHERE Operators

WHERE age BETWEEN 18 AND 30
WHERE name LIKE 'A%'       -- starts with A
WHERE name LIKE '%son'     -- ends with son
WHERE name LIKE '_ob'      -- _ = any char
WHERE status IN ('a', 'b')
WHERE email IS NOT NULL
WHERE NOT (age < 18)
WHERE a = 1 AND b = 2
WHERE a = 1 OR  b = 2
08

JOIN Types

All JOIN Types

-- INNER JOIN: only matching rows
SELECT u.username, o.total
FROM  users u
INNER JOIN orders o ON u.id = o.customer_id;

-- LEFT JOIN: all from left + matches
SELECT u.username, o.total
FROM  users u
LEFT JOIN orders o ON u.id = o.customer_id;
-- users with NO orders → o.total = NULL

-- RIGHT JOIN: all from right + matches
SELECT u.username, o.total
FROM  users u
RIGHT JOIN orders o ON u.id = o.customer_id;

-- FULL OUTER JOIN (not in MySQL)
SELECT * FROM a
FULL OUTER JOIN b ON a.id = b.id;

-- CROSS JOIN: every combination
SELECT * FROM colors CROSS JOIN sizes;

Self Join & USING

-- Self Join: table joins itself
SELECT e.name AS employee,
       m.name AS manager
FROM   employees e
LEFT JOIN employees m
  ON e.manager_id = m.id;

-- USING shorthand (same column name)
SELECT * FROM orders
JOIN customers USING (customer_id);
Rule of thumb: INNER JOIN when you only want matched records. LEFT JOIN when you want all from the left table, even with no match.

Aggregate Functions & GROUP BY

Aggregate Functions

COUNT(*)         -- total rows
COUNT(col)        -- non-null count
COUNT(DISTINCT col) -- unique count
SUM(col)          -- total
AVG(col)          -- average
MIN(col)          -- smallest
MAX(col)          -- largest
GROUP_CONCAT(col) -- join strings (MySQL)

GROUP BY + HAVING

-- Orders per customer (count ≥ 2)
SELECT   customer_id,
         COUNT(*) AS order_count,
         SUM(total) AS revenue
FROM     orders
GROUP BY customer_id
HAVING   COUNT(*) >= 2
ORDER BY revenue DESC;

-- WHERE filters rows BEFORE grouping
-- HAVING filters AFTER grouping

Subqueries & CTEs

Subqueries

-- Scalar subquery in SELECT
SELECT name,
  (SELECT COUNT(*) FROM orders
   WHERE customer_id = u.id) AS total_orders
FROM users u;

-- Subquery in WHERE
SELECT * FROM products
WHERE price > (SELECT AVG(price) FROM products);

-- EXISTS subquery
SELECT * FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders
  WHERE customer_id = u.id
);

CTE (Common Table Expression)

-- Clean, readable alternative to subqueries
WITH top_customers AS (
  SELECT   customer_id, SUM(total) AS spent
  FROM     orders
  GROUP BY customer_id
  HAVING   SUM(total) > 1000
),
vip AS (
  SELECT u.username, tc.spent
  FROM   top_customers tc
  JOIN   users u ON u.id = tc.customer_id
)
SELECT * FROM vip ORDER BY spent DESC;

Indexes & Performance

Creating Indexes

-- Single column index
CREATE INDEX idx_email
  ON users(email);

-- Unique index (prevents duplicates)
CREATE UNIQUE INDEX idx_username
  ON users(username);

-- Composite index
CREATE INDEX idx_name_status
  ON users(status, created_at);

-- Drop index
DROP INDEX idx_email ON users;

Index Rules

  • ✓
    Index columns used in WHERE, JOIN ON, ORDER BY
  • ✓
    Foreign key columns should almost always be indexed
  • ✓
    Composite indexes — order matters; leads with most selective column
  • ✗
    Don't over-index — each index slows INSERT/UPDATE/DELETE
  • ✗
    Avoid indexing low-cardinality columns (e.g. boolean)
Use EXPLAIN SELECT ... to see if an index is being used by the query planner.

Transactions & ACID

ACID Properties

  • A
    Atomicity — all operations succeed, or all are rolled back
  • C
    Consistency — data moves from one valid state to another
  • I
    Isolation — concurrent transactions don't interfere
  • D
    Durability — committed data persists even after crashes

Transaction Syntax

START TRANSACTION;  -- or BEGIN;

UPDATE accounts
  SET balance = balance - 500
  WHERE id = 1;

UPDATE accounts
  SET balance = balance + 500
  WHERE id = 2;

-- If everything is OK:
COMMIT;

-- If something fails, undo all:
ROLLBACK;

-- Save partial progress
SAVEPOINT sp1;
ROLLBACK TO sp1;

Views & Window Functions

Views

-- Create a reusable named query
CREATE VIEW active_users AS
  SELECT id, username, email
  FROM   users
  WHERE  status = 'active';

-- Use it like a table
SELECT * FROM active_users;

-- Drop a view
DROP VIEW active_users;
Views don't store data. They are saved SQL queries executed fresh on each call.

Window Functions

-- ROW_NUMBER, RANK, DENSE_RANK
SELECT
  username,
  total,
  ROW_NUMBER()
    OVER(ORDER BY total DESC) AS row_num,
  RANK()
    OVER(ORDER BY total DESC) AS rank_pos
FROM users;

-- Running total with PARTITION
SELECT
  dept,
  salary,
  SUM(salary) OVER(
    PARTITION BY dept
    ORDER BY salary
  ) AS running_total
FROM employees;

Design Best Practices

  • ▸
    Always define a Primary Key on every table
  • ▸
    Use meaningful, consistent naming (snake_case)
  • ▸
    Normalize to 3NF; denormalize only for performance
  • ▸
    Index FK columns and frequently filtered columns
  • ▸
    Use transactions for multi-step write operations
  • ▸
    Prefer NOT NULL by default; allow NULL only intentionally
  • ▸
    Use DECIMAL for money, never FLOAT
  • ▸
    Run EXPLAIN to analyze slow queries
