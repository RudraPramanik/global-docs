01 — Overview

What is RAG?

LLMs are trained on fixed corpora — they can't reason about private, recent, or specialized data. RAG solves this by grounding generation in retrieved documents.

💡
Core idea: Instead of baking knowledge into model weights, RAG retrieves relevant documents at inference time and feeds them as context to the LLM — enabling factual, up-to-date, and domain-specific answers without fine-tuning.
🏗️

Why not fine-tuning?

Fine-tuning is expensive, slow to update, and poor at factual recall. RAG updates instantly — just swap the document store.

Cost-effective
🔒

Private Data

Your documents never leave your infrastructure. RAG works perfectly with on-premise vector stores and self-hosted LLMs.

Privacy
⚡

Always Current

Add new documents any time. Your knowledge base updates in real-time without touching the model weights.

Real-time
📎

Source Attribution

Every answer can cite its sources. Full traceability and auditability — critical for enterprise and regulated industries.

Auditable
🧠

In-Context Learning

Documents are injected directly into the LLM's context window, leveraging the model's reasoning ability on fresh data.

ICL
🔧

Modular Architecture

Swap embeddings, vector stores, LLMs, and retrieval strategies independently. Each component is replaceable.

Modular

03 — Quick Start

Get Running in 5 Minutes

Minimal working RAG pipeline using LangChain. Covers the naive baseline — a production starting point.

01

Install dependencies

Core stack: LangChain, OpenAI embeddings, Chroma vector store.

02

Load & chunk documents

Load your source documents and split into overlapping chunks for better semantic coverage.

03

Embed & index

Convert chunks to vectors and persist them in a local Chroma DB.

Retrieve & generate

At query time, embed the question, find top-k similar chunks, and prompt the LLM with context.

# Set your API key
export OPENAI_API_KEY="sk-..."
export LANGCHAIN_API_KEY="ls__..."  # for LangSmith tracing (optional)
python
import bs4
from langchain_community.document_loaders import WebBaseLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

# 1. Load documents
loader = WebBaseLoader(
    web_paths=("https://lilianweng.github.io/posts/2023-06-23-agent/",),
    bs_kwargs=dict(parse_only=bs4.SoupStrainer(
        class_=("post-content", "post-title")
    ))
)
docs = loader.load()

# 2. Split into chunks
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
splits = splitter.split_documents(docs)

# 3. Embed and store
vectorstore = Chroma.from_documents(
    documents=splits,
    embedding=OpenAIEmbeddings()
)
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 4}
)
python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# Prompt template
prompt = ChatPromptTemplate.from_template("""
You are an expert assistant. Use only the provided context.
If unsure, say you don't know.

Context: {context}

Question: {question}
""")

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# Format docs helper
def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# Query
question = "What is Task Decomposition?"
context_docs = retriever.invoke(question)
context = format_docs(context_docs)

answer = llm.invoke(
    prompt.invoke({"context": context, "question": question})
)
print(answer.content)
python
# Composable LCEL chain — the idiomatic LangChain way
rag_chain = (
    {
        "context": retriever | format_docs,
        "question": RunnablePassthrough()
    }
    | prompt
    | llm
    | StrOutputParser()
)

# Single call to query the entire pipeline
answer = rag_chain.invoke("What is Task Decomposition?")
print(answer)

# Stream responses for better UX
for chunk in rag_chain.stream("What agents does the paper mention?"):
    print(chunk, end="", flush=True)
04 — Indexing (Notebooks 1–9)

Indexing Strategies

How you chunk and embed your documents is the single biggest lever on RAG quality. Bad indexing can't be fixed by better retrieval.

⚠️
Engineering insight: Chunking strategy is the most underestimated component. The right chunk size and overlap depends entirely on your document type, query style, and embedding model's token window.
01
Recursive Character Text Splitting
Notebooks 1–4 · Baseline
Splits text hierarchically on \n\n → \n → space → char. The default and usually best starting point. Preserves semantic boundaries naturally.
PROS
  • Respects natural text structure
  • Simple to configure
  • Works across all document types
CONS
  • Ignores document semantics
  • Fixed size may break context
02
Semantic Chunking
Notebooks 5–6 · Advanced Indexing
Embeds each sentence and detects semantic boundaries via cosine similarity drops. Chunks form around coherent topics rather than arbitrary character counts.
PROS
  • Topic-coherent chunks
  • Better retrieval precision
CONS
  • Slower (embeds every sentence)
  • More expensive at indexing time
03
Multi-Vector Indexing (Parent-Child)
Notebooks 7–9 · Precision + Context
Index small child chunks for precise retrieval, but return their full parent document as context. Best of both worlds: surgical retrieval with rich generation context. Also supports indexing hypothetical question-answer pairs.
python
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore

parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000)
child_splitter  = RecursiveCharacterTextSplitter(chunk_size=400)

store     = InMemoryStore()  # or Redis for production
vectordb  = Chroma(embedding_function=OpenAIEmbeddings())

retriever = ParentDocumentRetriever(
    vectorstore=vectordb,
    docstore=store,
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)
PROS
  • Precise retrieval with full context
  • Handles long documents well
CONS
  • Requires dual storage backend
  • More complex indexing pipeline
05 — Retrieval (Notebooks 10–11)

Retrieval Techniques

Getting the right documents to the LLM is the entire game. These strategies address the fundamental query-document mismatch problem.

Multi-Query Retrieval

Generate N alternative phrasings of the query using an LLM. Run retrieval for each. Merge unique results. Solves vocabulary mismatch — your query words may not match how documents are written.

python
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(),
    llm=llm,
    # generates 3 query variants by default
)

# Automatically handles multi-query + deduplication
docs = retriever.invoke("What are the types of memory in agents?")

RAG-Fusion + RRF

Generate sub-queries, retrieve for each, then re-rank all results using Reciprocal Rank Fusion (RRF). Documents appearing in top positions across multiple result lists are promoted. Robust and query-model-agnostic.

python
from langchain_core.prompts import ChatPromptTemplate

def reciprocal_rank_fusion(results: list[list], k=60):
    scores = {}
    for docs in results:
        for rank, doc in enumerate(docs):
            key = doc.page_content
            if key not in scores: scores[key] = (0, doc)
            cur_score, _ = scores[key]
            scores[key] = (cur_score + 1 / (rank + k), doc)
    return [doc for _, doc in
            sorted(scores.values(), key=lambda x: x[0], reverse=True)]

Query Decomposition

Break complex questions into simpler sub-questions. Answer each independently, then synthesize. Two modes: sequential (each answer informs the next) or parallel (all answered independently then merged).

python
decomp_prompt = ChatPromptTemplate.from_template("""
Break this question into 3 simpler sub-questions.
Return as a Python list.

Question: {question}
""")

decomp_chain = (
    decomp_prompt
    | ChatOpenAI(temperature=0)
    | StrOutputParser()
    | (lambda x: x.strip().split("\n"))
)

HyDE — Hypothetical Document Embeddings

Instead of embedding the short question, ask the LLM to generate a hypothetical full answer, then embed that. The embedding of a document-like answer is much closer in vector space to real documents than a short query.

python
hyde_prompt = ChatPromptTemplate.from_template("""
Write a short paragraph that would answer this question.
Don't include 'I don't know'. Write as if you're an expert.

Question: {question}
""")

hyde_chain = (
    hyde_prompt
    | ChatOpenAI(temperature=0)
    | StrOutputParser()
)

# Use the HyDE text for retrieval instead of raw question
hyde_retriever = hyde_chain | retriever
06 — Generation (Notebooks 12–14)

Generation Patterns

How you pass context to the LLM dramatically affects answer quality. These patterns address common failure modes.

📍

Contextual Compression

After retrieval, a secondary LLM extracts only the relevant portions from each chunk. Reduces noise, focuses generation on signal. Trades latency for precision.

Notebook 12
🎯

Step-Back Prompting

Ask the LLM to first think about the general principle, then answer the specific question. Reduces over-literal retrieval on nuanced queries.

Notebook 13
🔗

CRAG — Corrective RAG

Grade retrieved documents. If retrieval quality is low, fall back to web search or reformulate the query. Adds a self-correction loop.

Notebook 14
✅
Context placement matters: Studies show LLMs recall information from the start and end of context windows better than the middle. Put your most important chunks first or use reranking to surface the best ones.
python — Contextual Compression
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

compressor = LLMChainExtractor.from_llm(llm)

compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=retriever,
)

# Returns only the most relevant sentences from each retrieved doc
compressed_docs = compression_retriever.invoke(
    "What is the role of the planner in LLM agents?"
)
07 — Advanced RAG (Notebooks 15–18)

Advanced Techniques

Self-RAG, agentic routing, and adaptive pipelines — the frontier of RAG engineering.

15
Self-RAG — Reflective Retrieval
Adaptive retrieval with relevance grading
The LLM grades its own retrieved documents for relevance before generating. If docs are irrelevant, it reformulates the query and retrieves again. After generating, it grades the answer for hallucination and grounds answers in evidence. Requires a grading LLM call at each step.
16
Routing — Logical & Semantic
Direct queries to the right data source
Classify incoming queries and route them to the appropriate retriever (vector store, SQL DB, code search, API). Semantic routing embeds the query and compares to prompt embeddings. Logical routing uses an LLM classifier with structured output.
python — Structured Routing
from pydantic import BaseModel
from typing import Literal

class RouteQuery(BaseModel):
    datasource: Literal["vectorstore", "websearch", "sql"]

router = llm.with_structured_output(RouteQuery)

def route(query: str) -> str:
    result = router.invoke(
        f"Route this query to the right source: {query}"
    )
    return result.datasource
17
Query Construction
Convert natural language to structured queries
Translate natural language questions into structured metadata filters for your vector store. Ask "find papers about RAG from 2023 by LangChain" → extract filter={"year": 2023, "author": "langchain"}. Also covers text-to-SQL and text-to-Cypher.
18
Agentic RAG with LangGraph
Stateful, looping, tool-using RAG agents
Build RAG systems as state machines using LangGraph. Nodes for retrieval, grading, generation, and decision-making. Edges encode conditional logic (retry if docs irrelevant, fall back to web search, loop until answer is grounded). Enables complex multi-step reasoning and adaptive pipelines.
python — LangGraph RAG Agent Skeleton
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class AgentState(TypedDict):
    question: str
    documents: List
    generation: str
    retries: int

workflow = StateGraph(AgentState)

# Add nodes
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("grade_docs", grade_documents_node)
workflow.add_node("generate", generate_node)
workflow.add_node("transform_query", rewrite_query_node)
workflow.add_node("web_search", web_search_node)

# Conditional edges
workflow.add_conditional_edges(
    "grade_docs",
    decide_to_generate,  # returns "generate" or "transform_query"
)

workflow.set_entry_point("retrieve")
app = workflow.compile()

python — RAGAS Evaluation
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall
)
from datasets import Dataset

eval_data = {
    "question": ["What is Task Decomposition?"],
    "answer": ["Task Decomposition breaks complex tasks..."],
    "contexts": [[doc.page_content for doc in retrieved_docs]],
    "ground_truth": ["Task Decomposition is the process of..."],
}

result = evaluate(
    dataset=Dataset.from_dict(eval_data),
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)
print(result.to_pandas())
09 — Production

Production Checklist

Everything you need before shipping a RAG system to production. Click items to track your progress.

📑 Indexing

0 / 5
  • ✓Choose and benchmark chunk size for your domain (try 256, 512, 1024)
  • ✓Add chunk_overlap (10–20% of chunk_size) to avoid boundary artifacts
  • ✓Store rich metadata (source URL, section title, date, doc type) with each chunk
  • ✓Test multiple embedding models (text-embedding-3-small vs large vs BGE)
  • ✓Set up incremental indexing — re-embed only changed documents

🔍 Retrieval

0 / 5
  • ✓Benchmark top-k (3, 5, 10) and choose based on context window budget
  • ✓Add hybrid search (dense + BM25 sparse) for keyword-sensitive queries
  • ✓Implement a cross-encoder reranker (Cohere Rerank, BGE-Reranker) post-retrieval
  • ✓Log all retrieval misses — queries that returned 0 relevant docs
  • ✓Add metadata filtering so users can scope queries to specific sources/dates

✨ Generation & Prompting

0 / 5
  • ✓Instruct the model to say "I don't know" when context is insufficient
  • ✓Always include source citations in the generated answer
  • ✓Set temperature=0 for factual Q&A, >0 for creative/summarization tasks
  • ✓Test with adversarial queries: out-of-scope, contradictory, ambiguous
  • ✓Implement answer streaming for better perceived latency

🚀 Infra & Ops

0 / 5
  • ✓Trace all RAG calls with LangSmith or similar for debugging
  • ✓Add caching layer for repeated queries (Redis, semantic cache)
  • ✓Measure and set P95 latency budgets per component
  • ✓Implement a feedback loop — thumbs up/down feeds back into eval set
  • ✓Set up offline eval suite with golden Q&A pairs; run on every deploy
10 — Reference

Strategy Comparison

Choose the right technique for your use case. Latency and cost are relative estimates.

Technique When to Use Latency Cost Complexity
Naive RAG Prototypes, simple Q&A on clean docs Low Low Low
Multi-Query Vocab mismatch, diverse query styles Med Med Low
HyDE Short ambiguous queries, knowledge-dense docs Med Med Low
Parent-Child Long documents, need full context Low Low Med
RAG-Fusion + RRF High-precision needs, diverse docs Med Med Med
CRAG Low-quality or sparse knowledge bases High High Med
Self-RAG Mission-critical accuracy, hallucination-sensitive High High High
Agentic (LangGraph) Multi-step, multi-source, complex reasoning High High High
⚠️
Start simple: Always begin with Naive RAG + a proper eval suite. Upgrade to advanced techniques only when you can measure the improvement. Premature optimization in RAG is expensive and often counterproductive.
11 — Cookbook

Code Cookbook

Production-ready snippets for common RAG engineering tasks.

python — Hybrid Search (Dense + BM25)
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

# Sparse (keyword) retriever
bm25_retriever = BM25Retriever.from_documents(splits)
bm25_retriever.k = 5

# Dense (semantic) retriever
dense_retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5}
)

# Ensemble — 40% BM25, 60% semantic
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, dense_retriever],
    weights=[0.4, 0.6]
)
python — Cohere Reranking
from langchain.retrievers.document_compressors import CohereRerank
from langchain.retrievers import ContextualCompressionRetriever

compressor = CohereRerank(
    model="rerank-english-v3.0",
    top_n=3,  # return only top 3 after reranking
)

rerank_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=ensemble_retriever,
)

# Returns 3 best docs from potentially 10 retrieved
final_docs = rerank_retriever.invoke("your query")

from fastapi import FastAPI

python — Semantic Caching
from langchain.cache import RedisSemanticCache
from langchain.globals import set_llm_cache

# Cache semantically similar queries
# "What is RAG?" ≈ "Define RAG" → same cached result
set_llm_cache(RedisSemanticCache(
    redis_url="redis://localhost:6379",
    embedding=OpenAIEmbeddings(),
    score_threshold=0.2  # lower = more aggressive caching
))

# All subsequent LLM calls automatically use the cache
# First call: ~800ms. Cached call: ~5ms
