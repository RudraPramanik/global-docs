Logs — the event record

A log is a timestamped record of something that happened. Think of it as the system's diary. Every error, every user action, every database call can produce a log entry. The key insight: logs are discrete events — they fire once, when something occurs.

# Bad — plain text log (hard to query, hard to parse) 2024-01-15 14:32:01 ERROR Failed to process request for user 42
# Good — structured JSON log (queryable, filterable, parseable) { "timestamp": "2024-01-15T14:32:01.234Z", "level": "ERROR", "logger": "rag_service", "message": "LLM call failed", "user_id": "u_abc123", "workspace_id": "ws_xyz", "latency_ms": 4821, "error": "timeout after 5000ms" }
Use when
Debugging a specific error
Auditing what a user did
Recording rare/important events
Capturing error messages + stack
Avoid for
High-frequency per-request stats
Aggregated counts (use metrics)
Request flow visibility (use traces)

Traces — the request journey

A trace follows a single request as it travels through your system. Each step is a span. Spans have a parent-child relationship, forming a tree. This tells you exactly where time was spent inside a single request — retrieval took 200ms, LLM took 1800ms, etc.

# A trace = one request. Spans = steps inside it. Trace: rag.answer (total: 2.1s) ├─ span: retrieval [0ms → 210ms] chunks: 8 ├─ span: context_building [210ms → 240ms] used: 5 └─ span: llm_generation [240ms → 2100ms]tokens: 842
Use when
Pinpointing slow steps in a request
Seeing LLM token usage / cost
Debugging AI pipeline quality
Cross-service request flow
Avoid for
System-wide health at a glance
Historical trend analysis
Alerting (use metrics instead)

Metrics — the aggregate numbers

Metrics are numeric measurements collected over time. Unlike logs (one event = one record), metrics are aggregated — they tell you "500 requests in the last minute" not "here's each request". They're cheap to store, fast to query, and ideal for dashboards and alerting.

# Counter — only goes up. Use for totals. http_requests_total{status="200"} → 45821 # Gauge — goes up and down. Use for current state. memory_usage_bytes → 142000000 # Histogram — measures distributions. Use for latency. request_duration_seconds_bucket{le="0.5"} → 3821 request_duration_seconds_bucket{le="1.0"} → 4512
# PromQL: rate of requests per second (last 5 min) rate(http_requests_total[5m]) # Error rate rate(http_requests_total{status=~"5.."}[5m]) # P95 latency histogram_quantile(0.95, rate(request_duration_seconds_bucket[5m]) )
Use when
Trending over time (hours, days)
Alerting thresholds
Capacity planning
SLA / uptime dashboards
Avoid for
Debugging a specific request
Seeing what a user did
Understanding per-call breakdown

How they work together

A real debugging scenario: users report the AI chat is slow. Here's how each pillar contributes.

1
Metric  Alert fires: P95 latency > 3s
Grafana dashboard shows a spike at 14:30. You know something is wrong and roughly when. But not why or where.
2
Trace  Open Langfuse, filter by that time window
You see that the llm_generation span is taking 4.2s. The retrieval span is normal (180ms). Now you know where: the LLM call itself.
3
Log  Search logs for the affected requests
Filter by the trace's request_id. You find a warning: "token budget exceeded, truncating context". The context was too large, causing the LLM to process more tokens than expected.
