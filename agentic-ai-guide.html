01

Tokens, tokenization & context windows

The fundamental unit of everything LLMs consume and produce. Every cost, latency, and limit flows from tokens.

What is a token
~4 chars · ~0.75 words
Sub-word pieces produced by BPE or SentencePiece. "unbelievable" → 3–4 tokens. Code is token-dense due to symbols and whitespace. Numbers can be 1 token each or many.
Context window
Input + output combined
The maximum tokens visible in one API call. Includes system prompt, history, and generation. Range: 8K–1M+ tokens. Exceeding it truncates or throws an error.
Tokenization variance
Language matters a lot
CJK (Chinese/Japanese/Korean) scripts use more tokens per word than English. Arabic, Hindi similarly. Always test token counts in your target language — pricing can be 2–3× higher.
Prompt caching
5–10× cheaper reads
Repeated prefix tokens (system prompts, long docs) cached server-side. Cache hits billed at ~10% of normal input cost. Essential for document agents.
⚠
Agentic cost multiplier: A multi-step agent re-sends the full context each turn. A 10-step loop with a 20K-token context = 200K input tokens total. Design short contexts and summarize aggressively from day one.

Cost formula: (input_tokens × input_$/1M) + (output_tokens × output_$/1M). Output tokens are always 3–5× more expensive. Optimize by reducing output length first — specify response format and length constraints in your prompt.

03

Closed vs open-weight models

The choice shapes your entire infra, compliance posture, and cost trajectory. Most mature systems end up hybrid.

☁ Closed / API — choose when…
  • Zero infra overhead required
  • Need frontier capability immediately
  • API volume is moderate (<$10K/mo)
  • Data leaving your network is acceptable
  • Need SLAs, support contracts, and uptime guarantees
  • Rapid experimentation without provisioning
🖥 Open-weight (self-hosted) — choose when…
  • Data must never leave your infrastructure
  • Fine-tuning on proprietary domain data needed
  • Volume is very high (>$50K/mo equivalent)
  • Model version pinning forever required
  • Regulatory requires explainability or on-prem
  • EU/GDPR mandates in-region processing
DimensionClosed APIOpen-weight self-hosted
LatencyShared infra; variable under loadDedicated; predictable and tunable
Throughput scalingImmediate — just increase rate limitsRequires GPU provisioning (days–weeks)
Fine-tuningLimited (OpenAI FT, Claude not yet)Full LoRA / QLoRA / full fine-tune
Data privacyData transits provider networkAir-gapped deployment possible
Model versionProvider updates; can break promptsYou pin the exact checkpoint forever
Cost at scaleLinear with token volumeFixed GPU cost; amortizes with volume
Breakeven (rough)~500M–2B tokens/month vs. single H100 server ($2–4K/mo)
🏗
Hybrid pattern (recommended): Closed API for experimentation, low-volume high-capability tasks, and customer-facing premium features. Open-weight self-hosted for high-volume, simple, or data-sensitive tasks. Migrate routes as volume grows past breakeven.
04

Sampling parameters

These control how the model selects the next token from the probability distribution. Wrong settings cause hallucinations, repetition, or incoherence.

⚡ Live temperature demo — "The capital of France is…"
temperature 0.0
Paris. (deterministic — always the same answer, ideal for agents)
temperature
Range: 0.0 – 2.0
Scales the probability distribution before sampling. High = more random, low = more deterministic. Set 0 for agentic tasks (reproducibility), 0.7–1.0 for creative generation. Never use >1.0 in production agents — hallucinations spike dramatically.
top_p
Range: 0.0 – 1.0
Nucleus sampling: only sample from the smallest set of tokens whose cumulative probability exceeds p. Cuts off the long tail of unlikely tokens. Prefer top_p or temperature — setting both creates unpredictable interactions. Typical: 0.9 for creative tasks.
top_k
Range: 0 (off) – vocab size
Only consider the top k tokens at each step. Simpler than top_p. Common in open-weight inference servers (llama.cpp, vLLM, Ollama). Set 40–80 for creative tasks, 0 (disabled) for structured extraction and agents. Not available on all APIs.
frequency_penalty
Range: –2.0 to 2.0
Penalizes tokens proportional to how many times they've already appeared. Reduces word-level repetition in long outputs. Good for reports and narratives. Use 0.3–0.7. Avoid in code agents — penalizes legitimate identifier reuse.
presence_penalty
Range: –2.0 to 2.0
Flat penalty on any token that appeared at all — encourages topic diversity more than frequency_penalty. Good for brainstorming (forces exploration of new ideas). Bad for code generation (avoids reusing correct identifiers).
max_tokens
Range: 1 – context_limit
Hard cap on output tokens. Model stops generating when limit is reached, even mid-sentence. Always set this — it prevents runaway costs and timeouts. For agentic steps: 512–2K. For long documents: 4K–8K. Check stop_reason = "max_tokens" to detect truncation.
🎯
Agent defaults: temperature=0, top_p=1 (disabled), top_k=0 (disabled), no penalties, max_tokens=1024–2048. These give reproducible, cost-predictable outputs. Only deviate when you have a specific, tested reason to.
05

Streaming vs non-streaming responses

Same model, same cost — but radically different UX and architecture patterns.

Non-streaming
Request → wait → full response
Response arrives as one JSON blob after full generation. Simpler to implement. Good for batch pipelines, tool chains where next step needs complete output, background jobs. Risk: timeout on long generations (>60s).
Streaming (SSE)
Tokens arrive as generated
Server-Sent Events. First token in ~300ms–1s. Essential for chat UIs, long document generation, anything users wait for. Reduces perceived latency by 10–50×. Slightly more complex error handling.
ConcernStreamingNon-streaming
Time to first byte~300ms–1sFull generation time (5–120s)
Timeout riskLow — connection kept alive during genHigh for long outputs on default timeouts
Error handlingHarder — failures mid-stream require retry logicSimple — check HTTP status code
CostIdentical — billed on same token counts
Tool use handlingMust buffer full tool_use block before callingSimple — parse complete response at once
JavaScript · Streaming (Anthropic SDK) 
const stream = await client.messages.stream({
  model: 'claude-sonnet-4-20250514',
  max_tokens: 1024,
  messages: [{ role: 'user', content: prompt }]
});

// Stream text to UI in real-time
for await (const chunk of stream) {
  if (chunk.type === 'content_block_delta' &&
      chunk.delta.type === 'text_delta') {
    ui.append(chunk.delta.text);
  }
}

const finalMsg = await stream.finalMessage();
console.log(finalMsg.usage); // {input_tokens: N, output_tokens: M}
⚠
Agentic streaming gotcha: When streaming with tool use, you must buffer the complete tool_use block before invoking the function — you cannot call a tool with partial JSON arguments. Stream text content to users immediately; accumulate tool_use blocks silently until input_json_delta events complete.

Structured outputs

Getting models to return machine-readable data reliably is one of the most critical skills in agentic engineering.

MethodHow it worksReliabilityWhen to use
Prompt-only
Ask for JSON in prompt		 Instruct model to "respond in JSON". Parse output. Low–medium Prototyping only. Always add try/catch + strip markdown fences.
JSON mode
OpenAI / Gemini API param		 API forces syntactically valid JSON at token level. High (syntax) Valid JSON guaranteed; schema adherence is NOT. Fields can still be wrong types or missing.
Schema-constrained
OpenAI structured_outputs		 Provide JSON Schema; output grammar-constrained to match exactly. Very high Production extraction. Slight first-call latency for schema compilation. Subset of JSON Schema only.
Tool-as-output
All providers		 Define a tool with input_schema; force model to "call" it with the data you want. Very high Most reliable cross-provider method. Works for extraction elegantly. The Claude-recommended approach.
JSON · Tool-as-structured-output pattern (Claude) 
{
  "tools": [{
    "name": "extract_invoice",
    "description": "Extract structured invoice data from document",
    "input_schema": {
      "type": "object",
      "properties": {
        "invoice_number": { "type": "string" },
        "total_amount":   { "type": "number" },
        "currency":       { "type": "string", "enum": ["USD","EUR","GBP"] },
        "line_items": {
          "type": "array",
          "items": {
            "type": "object",
            "properties": {
              "description": { "type": "string" },
              "amount":      { "type": "number" }
            },
            "required": ["description", "amount"]
          }
        }
      },
      "required": ["invoice_number", "total_amount"]
    }
  }],
  "tool_choice": { "type": "tool", "name": "extract_invoice" }
}
🔍
Always validate semantically: Even schema-constrained outputs can have semantically wrong values (negative amounts, future dates labeled as past, wrong currency codes). Run business-logic validation after schema validation — they are separate concerns.
07

Function calling / tool use mechanics

The core primitive of agentic systems. The model decides when to call a tool and with what arguments — your code executes it.

1
Define tools with JSON Schema
Pass an array of tool definitions to the API. Each has a name, description (critical — model reads this to decide when to use it), and input_schema. Write descriptions as if explaining to a smart colleague what the function does and when to use it.
2
Model returns tool_use block
When the model decides to call a tool, stop_reason = "tool_use" and the response content includes a tool_use block with id, name, and input (your validated arguments). Multiple tools can be called in one response.
3
Execute and return result
Run the function with the provided arguments. Return the result as a tool_result block in the next user message, referencing the tool_use_id. Include errors clearly — the model will adapt its plan accordingly.
4
Model continues or ends
The model receives tool results, reasons about them, and either continues (calls more tools) or returns a final text response (stop_reason = "end_turn"). Loop until end_turn or your max iteration guard fires.
JavaScript · Agentic loop skeleton 
async function agentLoop(messages, tools, maxIter = 20) {
  let iter = 0;

  while (iter++ < maxIter) {
    const res = await callModel({ messages, tools });

    // Add assistant turn to history
    messages = [...messages, { role: 'assistant', content: res.content }];

    if (res.stop_reason === 'end_turn') return res;

    if (res.stop_reason === 'tool_use') {
      const toolBlocks = res.content.filter(b => b.type === 'tool_use');

      // Execute tools in parallel for speed
      const results = await Promise.all(
        toolBlocks.map(async block => ({
          type: 'tool_result',
          tool_use_id: block.id,
          content: JSON.stringify(await executeTool(block.name, block.input))
        }))
      );

      messages = [...messages, { role: 'user', content: results }];
    }
  }

  throw new Error('Max iterations reached');
}
tool_choice optionBehaviorUse case
"auto"Model decides when to call toolsOpen-ended agents, conversational assistants
{"type":"tool","name":"X"}Forces model to call specific toolStructured extraction, guaranteed schema output
"none"Disables tool use for this turnSummarization step before committing to action
"required"Must call at least one toolAction-oriented agents that must do something
⚡
Parallel tool execution: Models can request multiple tool calls in a single response. Always handle an array of tool_use blocks — single-tool assumption is a common production bug. Execute independent tools in parallel with Promise.all to halve latency.

💰
Image token cost: A 1080p image costs ~1,500–3,000 tokens. Always resize before sending — scale to <768px on the short side for most tasks. For PDFs with many pages, extract text first (PyMuPDF, pdfplumber) and use native PDF mode only when visual layout or tables matter critically.
Vision agent patterns
Screenshot → model → structured action (UI automation). Chart image → model → extracted data table. Receipt photo → model → invoice JSON. Diagram → model → text description.
Audio pipeline (Claude)
Audio file → Whisper (transcription) → Claude (understanding, summarization, extraction). Chunk audio >25MB. For live voice: use GPT-4o Realtime or Gemini Live APIs.
URL vs base64
Most providers support both. URL is smaller payload but requires the image to be publicly accessible. Base64 works offline and for private assets but inflates request size ~33%.

Embedding models
Text → dense vector
Convert text chunks to fixed-size vectors (768–3072 dims). Semantically similar text has high cosine similarity. Used for ANN (approximate nearest neighbor) search. Speed: thousands of queries/sec. Recall: ~70–85%.
Reranker models (cross-encoders)
Query + document → relevance score
Take a (query, document) pair and output a relevance score. Far more accurate than embeddings (sees both together) but too slow for full-corpus search. Run after ANN retrieval to reorder top-K results. Biggest quality gain in RAG.

JavaScript · Two-stage RAG retrieval 
async function retrieveContext(query) {
  // Stage 1: Fast ANN search (top-50 candidates)
  const queryVec   = await embedder.embed(query);
  const candidates = await vectorDB.search(queryVec, { topK: 50 });

  // Stage 2: Reranker (picks best 8 from 50)
  const reranked = await reranker.rank(query, candidates);
  const context  = reranked.slice(0, 8);

  // Stage 3: Generate with high-quality context
  return llm.generate({ context, query });
}

// Skipping the reranker is the #1 RAG quality mistake
📊
Chunking strategy matters: Fixed-size chunks (512 tokens, 10% overlap) are the baseline. For better results: use semantic chunking (split on topic shifts), or parent-child retrieval (embed summaries, return full sections). Chunk size is a hyperparameter — tune it for your domain.

Reasoning models vs standard models

Reasoning models spend compute generating a hidden chain-of-thought before answering. Not always better — knowing when to use each saves money and latency.

DimensionStandard modelReasoning model (o1, R1, extended thinking)
LatencyFast — 1–10sSlow — 10s to 3+ min
CostNormal token pricing5–20× more (thinking tokens billed)
Math / formal logicGoodSignificantly better — competition math, proofs
Hard code debuggingGoodBetter on multi-step bugs needing full trace
Instruction followingExcellentSometimes worse — can overthink simple tasks
Agentic tool loopsExcellentSlower per step; not always higher quality
Streaming UXSmooth token-by-tokenLong silent pause then output burst
Available modelsClaude Sonnet/Haiku, GPT-4o, Gemini Flasho1, o3, DeepSeek-R1, Claude + extended_thinking
Use reasoning models when…
→ Hard math, formal proofs, verification
→ Code with subtle multi-step bugs
→ Long-horizon planning (strategy, architecture)
→ Legal/medical analysis requiring deep reasoning
→ One-shot high-stakes decisions (latency OK)
→ Benchmarkable tasks where quality is paramount
Use standard models when…
→ Real-time chat (latency is everything)
→ High-volume agentic loops (cost is everything)
→ Simple extraction, classification, summarization
→ Tool use with many rapid sequential steps
→ Streaming responses to users
→ Tasks where thinking doesn't change the answer

🧠
Budget token tuning: Start with 2,000–4,000 thinking tokens. Increase in steps until quality plateaus — this is your optimal budget for that task type. More budget doesn't always help: simple tasks see diminishing returns above ~4K, while complex multi-step proofs improve up to the max.
OpenAI o-series
o1, o3, o4-mini
Reasoning is hidden (no visible CoT). o3 is frontier-level for hard STEM. o4-mini is cost-efficient reasoning. Cannot combine with streaming on all endpoints. Set reasoning_effort: low / medium / high.
DeepSeek R1
Open-weight reasoning
Matches o1 on AIME/MATH benchmarks. Full weights downloadable. Can be self-hosted with SGLang or vLLM. Visible chain-of-thought (unlike o1). Very cheap on hosted API ($0.55/$2.19 per 1M in/out).
Hybrid routing
Smart task routing
Classify tasks: hard reasoning (math, complex planning) → reasoning model. Everything else → standard model. Even 5% of requests going to reasoning can dominate your cost — measure first.
