01

Embeddings

Q1 What is an embedding?
concept

An embedding is a fixed-length list of numbers (a vector) that represents the meaning of a piece of text in a high-dimensional space. Think of it as a coordinate: instead of "latitude, longitude" in 2D space, you get 1536 numbers in 1536-dimensional semantic space.

Core idea
A model trained on billions of text samples learns that certain words/phrases "live near" each other. "Dog" and "puppy" end up close. "Dog" and "quantum physics" end up far apart.

The model outputs these coordinates by passing your text through its neural network and reading the final hidden layer's activation values. OpenAI's text-embedding-3-small outputs 1536 floats; text-embedding-3-large outputs 3072.

Python · OpenAI SDK
from openai import OpenAI

client = OpenAI()

response = client.embeddings.create(
    input="The dog sat on the mat",
    model="text-embedding-3-small"
)

vector = response.data[0].embedding
# → [0.021, -0.045, 0.108, ...]  (1536 floats)

Vectors are normalized to length 1 (unit sphere), which makes cosine similarity equivalent to a simple dot product — very fast to compute.

Q2 Why do semantically similar texts produce similar vectors?
theory

The embedding model is trained using contrastive learning: pairs of sentences that mean the same thing are pulled together in vector space, while unrelated sentences are pushed apart. The model learns this from massive corpora with natural paraphrase pairs.

Distributional hypothesis
Words that appear in similar contexts carry similar meaning. "I ate a banana" and "I ate a mango" share identical context structure — the model learns both fruits belong in the same region of space.

The result: cosine similarity between two vectors tells you how semantically close they are. 1.0 = identical meaning, 0.0 = unrelated, −1.0 = opposite.

Q3 Why are embeddings generated offline instead of per request?
architecture

Embedding a large corpus at query time would be catastrophically slow and expensive. If you have 100,000 documents and a user sends a query, you cannot embed all 100k docs + 1 query simultaneously — that's 100,001 API calls per search.

❌ Per-request (naive)
  • Embed all docs every query
  • Latency = N × embed_time
  • Cost scales with traffic
  • Redundant work for unchanged docs
✓ Offline (correct)
  • Embed docs once at ingest time
  • Store vectors in vector DB
  • Query-time: embed query only (1 call)
  • Search is a fast ANN lookup

The ANN (Approximate Nearest Neighbor) search in a vector DB like Pinecone, Weaviate, or pgvector is sub-millisecond even for millions of vectors — far faster than re-embedding. The query embedding is the only API call needed at runtime.

Rule of thumb
Embed documents when they're ingested/updated. At query time, embed only the query (~1 token/ms) and ANN-search the pre-built index.
02

Embedding Pipeline

Q4 What happens from raw document → vector DB?
pipeline

The ingestion pipeline transforms raw text into searchable vectors in 6 discrete steps. Click each step to see details.

1
Load & Extract
PDF, HTML, DOCX → plain text
▾
Use a loader (LangChain, LlamaIndex, or custom) to extract clean text from source files. Strip boilerplate — headers, footers, nav menus. Preserve document metadata: source_url, title, last_modified. This metadata is stored alongside vectors and returned with search results.
2
Chunk
Split into overlapping segments
▾
Split text into ~512 token chunks with 10–20% overlap. Each chunk becomes one vector. Overlap ensures no semantic context is lost at boundaries. Store chunk_index and parent_doc_id so you can reconstruct full docs from chunks.
3
Hash
SHA-256 content fingerprint
▾
Compute SHA-256(chunk_text). Store alongside the vector. On subsequent runs, skip re-embedding if the hash matches — this makes incremental updates cheap and idempotent. Only re-embed chunks whose content actually changed.
4
Embed
Batch API call → float vectors
▾
Call the embedding API with batches of chunks (up to 2048 tokens per item, 2048 items per request). Collect the 1536-dim float vectors. Handle rate limits with exponential backoff. This is the most expensive step — only run it for new or changed chunks.
5
Upsert
Write to vector DB
▾
Write (vector, metadata) pairs to the vector DB. Use upsert semantics: insert if new, replace if exists (by chunk ID). The chunk ID is typically f"{doc_id}_{chunk_index}". For deleted documents, explicitly delete their chunk IDs from the index.
6
Query
Embed query → ANN search
▾
At query time: embed the user's question (1 API call), then run ANN search against the index to return the top-k most similar chunks. Pass those chunks as context to the LLM. This is the only pipeline step that runs at request time.
Q5 Where should chunking happen?
design

Chunking happens after extraction, before embedding. It's a pure text operation — no model calls needed. It should run in your ingestion worker, not at query time.

Why chunk at all?
Embedding models have a token limit (8191 for text-embedding-3). Long documents must be split. More importantly, smaller chunks produce more precise search results — a whole 50-page PDF vs a targeted 3-paragraph answer.

Chunking strategies, ranked by quality:

Strategy comparison
# 1. Recursive character splitter (best general purpose)
RecursiveCharacterTextSplitter(
    chunk_size=512,         # tokens
    chunk_overlap=64,       # 12% overlap
    separators=["\n\n", "\n", ".", " "]
)

# 2. Semantic chunking (best quality, slower)
#    Split when embedding similarity drops below threshold

# 3. Fixed token chunks (fastest, worst quality)
#    Blindly splits — ignores sentence boundaries

Always store the original text alongside the vector — you need it to return readable results. The vector is only for search; the text is what gets passed to the LLM.

Q6 Where should embedding happen?
infra

Embedding lives in your async ingestion worker — never in the web request handler. Reasons:

Ingestion worker responsibilities
Load documents → chunk → check content hash → skip unchanged → batch new chunks → call embedding API → upsert to vector DB → write hash cache

At query time, embedding happens once — for the user's question only. This single call must be fast; use async client and keep query text concise.

Async query embedding
async def embed_query(query: str) -> list[float]:
    resp = await async_client.embeddings.create(
        input=query,
        model="text-embedding-3-small"
    )
    return resp.data[0].embedding

# Then ANN search — sub-millisecond
results = vector_db.query(vector=await embed_query(q), top_k=5)
03

Updating Data

Q7 What happens when a document changes?
updates

When a document is updated, you cannot simply add new vectors — the old vectors for that document still exist in the index and will pollute search results with stale content. The correct update flow:

Update flow
1. Detect change (file watcher / webhook / scheduled scan)
2. Delete all existing chunk vectors for that doc_id
3. Re-extract and re-chunk the new version
4. Re-embed only changed chunks (via content hash)
5. Upsert new vectors
Incremental update logic
def update_document(doc_id, new_text):
    chunks = chunk_text(new_text)
    
    # Find which chunks actually changed
    changed = []
    for i, chunk in enumerate(chunks):
        new_hash = sha256(chunk)
        old_hash = hash_cache.get(f"{doc_id}_{i}")
        if new_hash != old_hash:
            changed.append((i, chunk, new_hash))
    
    # Remove obsolete chunk vectors (count may differ)
    vector_db.delete(filter={"doc_id": doc_id})
    
    # Embed only changed chunks, upsert all
    vectors = embed_batch([c[1] for c in changed])
    vector_db.upsert(vectors)

For deleted documents, call vector_db.delete(filter={"doc_id": doc_id}) and remove the hash cache entries. Leaving stale vectors is a common bug that causes ghost results in search.

Q8 Why use content hashes?
efficiency

Content hashing is the cheapest way to decide whether a chunk needs re-embedding. Instead of comparing full text strings or relying on last_modified timestamps (which can lie after a copy/restore), you hash the actual text bytes.

Why SHA-256 over timestamps
last_modified can change without content changing (file copy, metadata edit, git checkout). A content hash only changes when the bytes change. This makes the pipeline idempotent — safe to rerun anytime.

import hashlib

def content_hash(text: str) -> str:
    return hashlib.sha256(

Structured Outputs

Q9 Why are structured outputs better than free-form text?
reliability

Free-form text from an LLM is unpredictable — the model may change wording, formatting, or structure between runs. Any downstream code that parses it is fragile. Structured outputs enforce a contract: the model must return data that exactly matches your schema.

❌ Free-form text
  • Format varies between calls
  • Regex parsing breaks on edge cases
  • Hard to type-check downstream
  • LLM may add preamble text
  • Fields may be missing silently
✓ Structured output
  • Guaranteed JSON structure
  • Required fields always present
  • Types validated at model level
  • No parsing regex needed
  • Direct response.parsed access

OpenAI's structured outputs use a constrained decoding technique: the model's token logits are masked so only tokens that advance a valid JSON parse are allowed. This is stronger than simply asking the model to "respond in JSON" — it's enforced at the sampling level.

Q10 What problems do JSON schemas solve?
tooling

JSON Schema solves three problems: type safety (is this field a string or number?), completeness (required fields are never omitted), and validation (enum values, min/max, patterns). The schema is the contract between your code and the model.

Key schema tips: always set "additionalProperties": false to prevent extra fields, use description fields to guide the model, and keep schemas flat — deeply nested schemas produce more errors.

Reliability

Q11 How would you handle retries?
resilience

Use exponential backoff with jitter. Each retry waits longer than the last (2¹, 2², 2³... seconds) plus a random jitter to avoid thundering herd when many clients retry simultaneously.

▸ Interactive — exponential backoff simulator

click to simulate
Python · tenacity library (recommended)
from tenacity import (
    retry, stop_after_attempt,
    wait_exponential, retry_if_exception_type
)
from openai import RateLimitError, APIError

@retry(
    wait=wait_exponential(multiplier=1, min=4, max=60),
    stop=stop_after_attempt(5),
    retry=retry_if_exception_type((RateLimitError, APIError))
)
async def embed_with_retry(texts: list[str]):
    return await client.embeddings.create(
        input=texts,
        model="text-embedding-3-small"
    )
What to retry vs not
Retry: 429 (rate limit), 500/503 (server errors), network timeouts.
Do NOT retry: 400 (bad request — fix the input), 401 (auth — fix the key), 404 (wrong endpoint).
Q12 How would you handle rate limits?
throttling

OpenAI enforces two rate limit dimensions: RPM (requests per minute) and TPM (tokens per minute). Both can be hit independently. Your strategy must respect both.

Batching uses fewer requests (fewer RPM hits) and idle time allows token quota to refill

Strategies in order of effectiveness:

Rate limit playbook
1. Batch requests — pack as many texts per request as allowed (reduces RPM)
2. Asyncio semaphore — cap concurrent requests (e.g. max 10 in-flight)
3. Token counting pre-flight — track estimated TPM usage, sleep when near limit
4. Respect Retry-After header — on 429, wait exactly as long as the API says
5. Request higher limits — for production, contact OpenAI to raise your tier
Async semaphore + rate limit respecting
sem = asyncio.Semaphore(10)  # max 10 concurrent

async def embed_one_batch(batch):
    async with sem:
        try:
            return await embed_with_retry(batch)
        except RateLimitError as e:
            wait = int(e.response.headers.get(
                "Retry-After", 60
            ))
            await asyncio.sleep(wait)
            return await embed_with_retry(batch)
Q13 Why should embedding requests be batched?
performance

Each API call has fixed overhead: network round-trip (~50–200ms), TLS handshake amortization, server scheduling. Sending 100 texts in 1 request vs 100 individual requests saves ~99× the fixed overhead and reduces RPM consumption by 99×.

OpenAI batching limits
text-embedding-3-small/large: up to 2048 inputs per request, each input up to 8191 tokens. In practice, batch by token count (aim for ~100k tokens/request) rather than item count to avoid hitting the token limit.
Efficient batch embedding
def batch_embed(texts: list[str], batch_size: int = 100):
    all_embeddings = []
    for i in range(0, len(texts), batch_size):
        batch = texts[i : i + batch_size]
        resp = client.embeddings.create(
            input=batch,
            model="text-embedding-3-small"
        )
        # Results are in same order as input
        all_embeddings.extend(
            [d.embedding for d in resp.data]
        )
    return all_embeddings

Note: response items are guaranteed to be in the same order as input items — no need to sort by index.
