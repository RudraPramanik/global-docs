Core Design Principles

🎯
Single Responsibility

A class should have one, and only one, reason to change. Each module owns a single concern.

🔒
Open/Closed

Open for extension, closed for modification. Add new behavior without touching existing code.

🔄
DRY (Don't Repeat Yourself)

Every piece of knowledge must have a single, unambiguous representation in the system.

🧩
Separation of Concerns

Divide a program into distinct sections, each addressing a separate concern or responsibility.

📦
Encapsulation

Bundle data with the methods that operate on it. Hide internal state, expose behavior via interfaces.

🔀
Composition over Inheritance

Favor composing objects with desired behaviors over inheriting from base classes.

🚫
YAGNI

You Aren't Gonna Need It. Don't add functionality until it's actually needed.

⬇️
Dependency Inversion

Depend on abstractions, not concretions. High-level modules should not depend on low-level modules.

02

SOLID Principles

S
Single Responsibility Principle
A class has one job. UserService handles users, not emails. Split: EmailService sends emails. Bad: one "God class" does everything.
O
Open/Closed Principle
Extend behavior via new classes/interfaces, never by modifying working code. Use Strategy or Template Method patterns.
L
Liskov Substitution Principle
Subclasses must be substitutable for their base class without breaking the program. If Bird has fly(), don't make Penguin extend it.
I
Interface Segregation Principle
Many specific interfaces > one fat interface. Don't force classes to implement methods they don't need. Split IAgent into IExecutable, ISchedulable, etc.
D
Dependency Inversion Principle
Program to interfaces, not implementations. Inject dependencies. AgentRunner(ILLMClient llm) not AgentRunner(OpenAIClient llm).
03

LLD Design Process

💡
In interviews and real-world design, follow this structured approach. Rushing to code without analysis is the most common LLD mistake.

Clarify Requirements

Identify functional requirements (what it does) and non-functional (scale, performance, consistency). Ask: who are the actors? What are the primary use cases? What are the constraints?

Identify Core Entities & Objects

Extract nouns from requirements. These become your classes/interfaces. Group related data and behavior together. Assign clear responsibilities to each entity.

Define Relationships

Map associations (uses-a), aggregations (has-a), compositions (owns-a), and inheritance (is-a). Use UML notation to visualize. Prefer composition over deep inheritance.

Design Interfaces & Contracts

Define public APIs before implementation. Interfaces should be minimal, cohesive, and stable. Think about what each module publishes vs. hides.

Apply Design Patterns

Identify recurring problems and apply established patterns. Don't force patterns — let the problem guide the solution. Common: Factory, Strategy, Observer, Repository.

Validate Against SOLID & Edge Cases

Walk through use cases. Check: What if this module changes? Can I extend without modifying? Are there circular dependencies? Handle null, empty, error states.

Creational Patterns

🏭 Factory Method

Define an interface for creating an object, but let subclasses decide which class to instantiate.

interface IAgent { run(): void }

class AgentFactory {
  static create(type: string): IAgent {
    switch(type) {
      case 'research': return new ResearchAgent();
      case 'writer': return new WriterAgent();
      default: throw new Error('Unknown type');
    }
  }
}
Use when: object type determined at runtime

🔱 Abstract Factory

Create families of related objects without specifying their concrete classes.

interface ILLMFactory {
  createChat(): IChatModel;
  createEmbed(): IEmbedModel;
}
class OpenAIFactory implements ILLMFactory {
  createChat() { return new GPT4(); }
  createEmbed() { return new Ada002(); }
}
Use when: product families need consistency

🔐 Singleton

Ensure a class has only one instance and provide global access to it.

class TenantRegistry {
  private static instance: TenantRegistry;
  private constructor() {}
  
  static getInstance() {
    if (!TenantRegistry.instance)
      TenantRegistry.instance = new TenantRegistry();
    return TenantRegistry.instance;
  }
}
Use when: exactly one shared resource needed

🏗️ Builder

Construct complex objects step by step, separating construction from representation.

class AgentPipelineBuilder {
  private pipeline = new AgentPipeline();
  
  withLLM(llm: ILLMClient) {
    this.pipeline.llm = llm; return this;
  }
  withTools(tools: ITool[]) {
    this.pipeline.tools = tools; return this;
  }
  build() { return this.pipeline; }
}
Use when: complex objects with many optional parts

// DECORATOR PATTERN — wrapping agent with rate limiting + logging
interface IAgentExecutor {
  execute(task: Task): Promise<Result>;
}

class RateLimitedAgent implements IAgentExecutor {
  constructor(private inner: IAgentExecutor, private limiter: IRateLimiter) {}
  
  async execute(task: Task) {
    await this.limiter.acquire();       // check rate limit
    return this.inner.execute(task);  // delegate to real agent
  }
}

class LoggedAgent implements IAgentExecutor {
  constructor(private inner: IAgentExecutor, private logger: ILogger) {}
  
  async execute(task: Task) {
    this.logger.info(`Executing: ${task.id}`);
    const result = await this.inner.execute(task);
    this.logger.info(`Done: ${result.status}`);
    return result;
  }
}
// Usage: new LoggedAgent(new RateLimitedAgent(new BaseAgent(), limiter), logger)

Behavioral Patterns

📋 Strategy

Define a family of algorithms and make them interchangeable.

interface IRoutingStrategy {
  route(task: Task): IAgent;
}
class CapabilityRouter implements IRoutingStrategy {
  route(task) { /* match by skill */ }
}
class LoadBalancedRouter implements IRoutingStrategy {
  route(task) { /* pick least loaded */ }
}

👀 Observer

Define a one-to-many dependency so objects are notified automatically of state changes.

interface IEventListener {
  on(event: AgentEvent): void;
}
class AgentEventBus {
  listeners = new Map<string, IEventListener[]>();
  subscribe(type: string, l: IEventListener) { ... }
  publish(event: AgentEvent) { ... }
}

⛓️ Chain of Responsibility

Pass requests along a chain of handlers. Each decides to handle or pass on.

abstract class Middleware {
  protected next?: Middleware;
  setNext(m: Middleware) { this.next = m; }
  abstract handle(req: Request): void;
}
// Auth → RateLimit → Tenant → Handler

📝 Command

Encapsulate a request as an object. Supports undo, queuing, logging.

interface ICommand {
  execute(): Promise<void>;
  undo(): Promise<void>;
}
class RunAgentCommand implements ICommand {
  constructor(private agent: IAgent, private task: Task) {}
  execute() { return this.agent.run(this.task); }
  undo() { return this.agent.cancel(); }
}

🏛️ Template Method

Define skeleton of an algorithm in a base class, let subclasses fill in specific steps.

abstract class BaseAgent {
  async run(task: Task) {
    const ctx = await this.plan(task);
    const out = await this.act(ctx);
    return this.reflect(out);
  }
  abstract plan(t: Task): Promise<Context>;
  abstract act(c: Context): Promise<Output>;
}

📦 State

Allow an object to alter its behavior when its internal state changes.

interface IAgentState {
  handle(agent: AgentContext): void;
}
// States: Idle → Planning → Executing
//        → Reflecting → Done | Failed
class ExecutingState implements IAgentState {
  handle(agent) {
    agent.runTools();
    agent.setState(new ReflectingState());
  }
}

// Case Study

Multi-Tenant AI Agentic SaaS

Designing AgentOS — a platform where organizations deploy, orchestrate, and monitor custom AI agents with full tenant isolation, rate limiting, and extensible tool ecosystems.

CS·1

System Overview: AgentOS

🏢

Multi-Tenant

Each organization (tenant) gets isolated agents, data, usage quotas, and billing. Zero data bleed between tenants.

🤖

AI Agentic

Agents can plan, use tools, browse web, write code, call APIs, and reflect in multi-step ReAct loops.

☁️

SaaS Platform

Subscription-based. Self-serve onboarding. API-first. Marketplace for agent templates and tools.

CS·2

Requirements Analysis

✅ Functional Requirements

  • Tenants can create, configure, and deploy AI agents
  • Agents execute tasks using pluggable tool ecosystem
  • Multi-step reasoning via ReAct (Plan → Act → Observe → Reflect)
  • Tenant-level rate limiting and token quota management
  • Agent execution history, logs, and audit trails per tenant
  • Multi-LLM support (OpenAI, Anthropic, local models)
  • Webhook callbacks and async execution
  • Team management — users, roles, permissions per tenant
  • Marketplace — publish/install agent templates and tools

⚙️ Non-Functional Requirements

  • Strict data isolation between tenants (row-level security)
  • Extensible: add new agent types, tools, LLMs without code change
  • Horizontally scalable agent execution workers
  • Observability: structured logs, traces, metrics per tenant
  • Low-latency agent status updates (websocket / SSE)
  • Plugin system: 3rd-party tools without platform redeployment
  • Idempotent task execution with retry/backoff
  • Strong typing, validated at compile-time (TypeScript)
🔑
Key Design Insight: The hardest constraint is tenant isolation with extensibility. Every new feature (tools, LLMs, agent types) must work across all tenants without exposing one tenant's data to another.

BaseAgent — Template Method Pattern

abstract class BaseAgent implements IAgent {
  protected state: IAgentState = new IdleState();
  protected memory: ConversationMemory = [];
  protected tools: Map<string, ITool> = new Map();

  constructor(
    readonly id: string,
    readonly tenantId: string,
    protected llm: ILLMClient,         // injected: DIP ✓
    protected eventBus: IEventBus,     // injected: DIP ✓
    protected logger: ILogger          // injected: DIP ✓
  ) {}

  // TEMPLATE METHOD — skeleton algorithm
  async run(task: Task): Promise<AgentResult> {
    this.setState(new PlanningState());
    this.eventBus.publish({ type: 'agent.started', agentId: this.id, task });

    let maxSteps = 20, step = 0;
    while (step++ < maxSteps) {
      const plan = await this.plan();        // ABSTRACT — subclass fills
      if (plan.done) break;

      this.setState(new ExecutingState());
      const results = await this.act(plan);  // ABSTRACT — subclass fills

      this.setState(new ReflectingState());
      const next = await this.reflect(results); // ABSTRACT — subclass fills
      if (next.done) break;
    }

    this.setState(new DoneState());
    this.eventBus.publish({ type: 'agent.completed', agentId: this.id });
    return this.buildResult();
  }

  protected abstract plan(): Promise<PlanResult>;
  protected abstract act(plan: PlanResult): Promise<ActionResult[]>;
  protected abstract reflect(results: ActionResult[]): Promise<ReflectResult>;

  private setState(s: IAgentState) {
    this.state = s;
    this.logger.info(`[${this.id}] State: ${s.name}`);
  }
}

Tenant-Scoped Repository Proxy

// PROXY PATTERN — auto-injects tenantId, prevents cross-tenant data access
class TenantScopedAgentRepository implements IAgentRepository {
  constructor(
    private inner: IAgentRepository,
    private tenantId: string
  ) {}

  findById(id: string) {
    // Always scope to current tenant — cannot be bypassed
    return this.inner.findById(id, this.tenantId);
  }

  save(agent: AgentEntity) {
    if (agent.tenantId !== this.tenantId)
      throw new SecurityError('Cross-tenant write rejected');
    return this.inner.save(agent);
  }

  findByTenant() {
    return this.inner.findByTenant(this.tenantId); // locked to this tenant
  }
}

LLM Adapter Pattern

// ADAPTER — normalizes different LLM APIs to ILLMClient
class OpenAIAdapter implements ILLMClient {
  constructor(private client: OpenAI, private model: string) {}

  async complete(messages: Message[], opts: CompletionOpts) {
    const res = await this.client.chat.completions.create({
      model: this.model,
      messages: messages.map(toOpenAIFormat), // normalize
      temperature: opts.temperature ?? 0.7,
    });
    return fromOpenAIFormat(res);              // normalize out
  }

  async *stream(messages: Message[], opts: CompletionOpts) {
    const stream = await this.client.chat.completions.create({
      ...opts, messages: messages.map(toOpenAIFormat), stream: true
    });
    for await (const chunk of stream)
      yield chunk.choices[0]?.delta?.content ?? '';
  }
}

Quota & Rate Limit Design

class TenantQuotaManager {
  constructor(private redis: IRedisClient) {}

  async consumeTokens(tenantId: string, tokens: number): Promise<void> {
    const key = `quota:${tenantId}:${currentMonth()}`;
    const used = await this.redis.incrBy(key, tokens);
    const limit = await this.getLimit(tenantId);

    if (used > limit)
      throw new QuotaExceededError(`Tenant ${tenantId} exceeded token quota`);
  }

  async checkConcurrentAgents(tenantId: string): Promise<void> {
    const active = await this.redis.sCard(`active:${tenantId}`);
    const maxConcurrent = await this.getMaxConcurrent(tenantId);
    if (active >= maxConcurrent)
      throw new ConcurrencyError('Max concurrent agents reached');
  }
}

ReActAgent — Concrete Implementation

class ReActAgent extends BaseAgent {
  protected async plan(): Promise<PlanResult> {
    const systemPrompt = this.buildSystemPrompt(); // injects tools schema
    const response = await this.llm.complete([
      { role: 'system', content: systemPrompt },
      ...this.memory,
    ], { temperature: 0.2, tools: this.getToolsSchema() });

    return this.parsePlan(response); // {toolName, args, done}
  }

  protected async act(plan: PlanResult): Promise<ActionResult[]> {
    if (plan.done) return [];

    const tool = this.tools.get(plan.toolName);
    if (!tool) throw new Error(`Tool not found: ${plan.toolName}`);

    const ctx: ExecutionContext = { tenantId: this.tenantId, agentId: this.id };
    const result = await tool.execute(plan.args, ctx);

    this.eventBus.publish({ type: 'tool.executed', tool: plan.toolName, result });
    return [result];
  }

  protected async reflect(results: ActionResult[]): Promise<ReflectResult> {
    this.memory.push({
      role: 'tool',
      content: JSON.stringify(results)
    });
    const isDone = results.every(r => r.status === 'success') &&
                   this.taskSatisfied();
    return { done: isDone };
  }

  private buildSystemPrompt(): string {
    return `You are an AI agent. Available tools:\n
${[...this.tools.values()].map(t => `- ${t.name}: ${t.description}`).join('\n')}
Think step by step. Use tools when needed. Stop when task is complete.`;
  }
}

Tool Registry — Plugin System

// Extensible tool plugin system — add tools without changing agent code
class ToolRegistry {
  private tools: Map<string, ITool> = new Map();
  private tenantOverrides: Map<string, Set<string>> = new Map();

  register(tool: ITool): void {
    this.tools.set(tool.name, tool);
  }

  enableForTenant(tenantId: string, toolName: string): void {
    if (!this.tenantOverrides.has(tenantId))
      this.tenantOverrides.set(tenantId, new Set());
    this.tenantOverrides.get(tenantId)!.add(toolName);
  }

  getForTenant(tenantId: string): ITool[] {
    const enabled = this.tenantOverrides.get(tenantId) ?? new Set();
    return [...this.tools.values()].filter(t => enabled.has(t.name));
  }
}

// Built-in tools — each isolated, each independently testable
class WebSearchTool implements ITool {
  name = 'web_search';
  description = 'Search the web for current information';
  schema = { type: 'object', properties: { query: { type: 'string' } } };
  async execute(args: { query: string }, ctx: ExecutionContext) {
    // tenant-aware: rate limit per tenant, log per tenant
    return { status: 'success', data: await searchWeb(args.query) };
  }
}

Anti-Patterns to Avoid

🔴 God Class

One class that knows too much and does too much. Signs: 1000+ lines, methods in every domain, every other class depends on it.

// ❌ Bad: AgentManager does everything
class AgentManager {
  createAgent() {}    // creation
  chargeUser() {}     // billing!
  sendEmail() {}      // notifications!
  runSQL() {}         // DB access!
}

🔴 Shotgun Surgery

One change requires many small edits scattered across many classes. Sign of poor cohesion — related things are split up.

// ❌ Adding a new LLM requires
// editing: config.ts, factory.ts,
// router.ts, validator.ts, docs.md...
// ✅ Add one adapter class only

🔴 Leaky Abstraction

Internal details of a module bleed through its interface. Callers must know implementation details to use it correctly.

// ❌ Leaking SQL through interface
interface IUserRepo {
  findBySQLWhere(sql: string): User[];
  runRawQuery(q: string): any;
}

🔴 Deep Inheritance

3+ levels of inheritance creates fragile hierarchies. Changes to base class ripple unpredictably. Hard to test in isolation.

// ❌ Fragile hierarchy
Entity → BaseAgent → LLMAgent
  → ReActAgent → ResearchAgent
  → SpecializedResearcher

LLD Interview Checklist

📋 Before You Start

  • Ask: What are the primary use cases / actors?
  • Ask: Scale constraints? Extensibility requirements?
  • Ask: What's the most important entity / object?
  • State assumptions explicitly before coding
  • Draw relationships before writing any code

🏗️ During Design

  • Define interfaces before concrete classes (DIP)
  • Name classes clearly: Repository, Service, Factory, Strategy
  • Justify every design pattern you use
  • Handle edge cases: null, empty, concurrent, error
  • Think about extensibility: "What if we add X later?"

✅ Self-Review

  • Does each class have a single, clear responsibility?
  • Are dependencies injectable (not hard-coded)?
  • Can I add a new feature without modifying existing classes?
  • Are interfaces minimal and stable?
  • Is there any duplicated logic I can extract?

🗣️ Communication Tips

  • Think aloud — interviewers want your reasoning, not just code
  • Name the pattern you're applying and why
  • Draw UML diagrams — even rough ones help clarity
  • Discuss trade-offs, not just your chosen approach
  • Ask for feedback: "Does this seem too complex?"
