The context-efficient AI dev system
minimum tokens · maximum accuracy · zero context explosion
Core idea: AI coding tools (Cursor, Claude, Copilot) read your MD files as context. Fat, unstructured docs = token waste + hallucinations. Lean, layered docs = precise answers, faster, cheaper.
The problem
Without this system
AI reads nothing or everything, guesses conventions, breaks existing patterns, re-asks obvious questions, hallucinates imports and function signatures.
With this system
AI knows stack, patterns, file structure, what not to touch. Answers are precise on first attempt. Token usage drops 40–70%.

Guiding principles
  • Every line in an MD file costs tokens — write only what the AI needs to do its job
  • Never duplicate info — reference other files instead of copy-pasting
  • Keep CLAUDE.md under 150 lines — it's a map, not an encyclopedia
  • Domain-specific detail lives in separate docs, loaded only when relevant
  • Update docs after every major refactor — stale context is worse than no context
  • Never paste full code examples into MD files
  • Never describe how something works if the code is self-documenting
  • Never list every file — only document non-obvious structure
MD file tree
how to organize context documents in your project
Rule: one file per concern. Never one giant file. Load depth increases linearly — the AI can always ask for more context from a linked file, but it can't un-read bloat.
project-root/ ├── CLAUDE.md # ← always loaded · root map (≤150 lines) │ ├── docs/ │ ├── architecture.md # stack, layers, data flow │ ├── api-schema.md # endpoints, request/response shapes │ ├── db-schema.md # models, relationships, migrations │ ├── conventions.md # naming, patterns, style rules │ ├── tasks.md # current sprint / active work │ ├── langgraph.md # graph topology, node contracts │ └── env.md # env vars, secrets structure (NO values) │ ├── backend/ │ └── # FastAPI source — no MD needed if conventions.md exists │ ├── frontend/ │ └── # React source │ └── agents/ └── # LangGraph graphs & nodes
File loading strategy
Always loaded auto
CLAUDE.md — Cursor/Claude reads this on every session. Keep it as a pointer map. Reference the other files; don't inline their content.
On-demand manual / @mention
docs/*.md — Load these via @docs/architecture.md in Cursor or include in prompt context for Claude. Only load what's relevant to the task.
LangGraph projects: add docs/langgraph.md specifically for graph topology. AI hallucinates node/edge contracts without it. This single file prevents 80% of agent bugs.
CLAUDE.md — the root file
always loaded · kept under 150 lines · no prose essays
This is the AI's first impression of your project. It must answer: what is this? what stack? where are things? what are the hard rules? Then stop. Everything else belongs in linked docs.
Required sections (in order)
1. Project identity required
1–3 lines. Name, purpose, what it is NOT. This prevents the AI from solving the wrong problem.
2. Stack declaration required
Exact versions matter. AI uses version-specific syntax. Ambiguity here = wrong imports, deprecated APIs.
3. Directory map required
Only non-obvious structure. Don't list node_modules/ or __pycache__/. 5–10 lines max.
4. Hard rules required
Short bullets. What the AI must never do. E.g. "never edit migrations directly", "always use the repo pattern for DB access".
5. Docs map recommended
A list of your docs/ files with one-line descriptions. Tells the AI where to look for deeper context.
Annotated template
# Project: YourAppName ## What this is # A task management API + React frontend with AI agent workflows. # NOT: a general-purpose chatbot or CMS. ## Stack - Python 3.12 / FastAPI 0.111 / SQLAlchemy 2.0 (async) - PostgreSQL 16 (Alembic migrations) - React 18 / TypeScript 5 / Vite / TanStack Query v5 - LangGraph 0.2 / LangChain 0.2 / OpenAI GPT-4o ## Directory structure # Only non-obvious paths backend/app/ api/ # route handlers only — no business logic services/ # business logic layer repositories/ # all DB queries live here agents/ # LangGraph graphs and nodes frontend/src/ features/ # feature-sliced: each feature owns components + hooks lib/ # shared utilities, api client ## Hard rules - NEVER write raw SQL — use repository layer only - NEVER import DB session in route handlers — use Depends() - NEVER mutate state directly in React — use TanStack Query mutations - NEVER add logic to LangGraph nodes — keep nodes thin, logic in services - ALWAYS run `alembic revision --autogenerate` for schema changes ## Docs index - @docs/architecture.md — system layers and data flow - @docs/api-schema.md — all endpoints and response shapes - @docs/db-schema.md — models and relationships - @docs/conventions.md — naming, error handling, patterns - @docs/langgraph.md — graph topology and node contracts - @docs/tasks.md — current sprint work
architecture.md
system layers · data flow · integration points
Load this when adding new features, refactoring layers, or debugging cross-service issues. It prevents the AI from creating shortcuts that bypass your architecture.
What to include
Template
LangGraph section
Include these
  • Layer diagram as ASCII — request flow from HTTP → router → service → repo → DB
  • External services and how they're integrated (OpenAI, Redis, S3, etc.)
  • Authentication flow — where JWT is validated, who creates sessions
  • Background task approach — Celery, ARQ, FastAPI BackgroundTasks?
  • LangGraph: graph names, node names, what state flows between them
  • Frontend: how API calls are made — fetch wrapper, axios, generated client?
Never include
  • Full code implementations — reference file paths instead
  • Deployment infra details — separate devops.md if needed
  • Historical decisions — forward-looking only
## Request flow HTTP Request → FastAPI router (api/routes/) # validate input, auth check → Service layer (services/) # business logic, orchestration → Repository layer (repositories/) # ALL database access → PostgreSQL (async SQLAlchemy) ## Auth - JWT Bearer in Authorization header - Validated in `app/core/security.py` → `get_current_user` dependency - No auth logic in route handlers — always use Depends(get_current_user) ## Background tasks - ARQ with Redis — `app/workers/` - Enqueue in service layer, never in routes - Long AI tasks (>10s) always go to background worker ## Frontend data layer - Generated OpenAPI client at `frontend/src/lib/api-client/` - TanStack Query for all server state — no useState for fetched data - Optimistic updates pattern in `frontend/src/lib/mutations.ts` ## AI agent layer - See @docs/langgraph.md for graph topology - All agents called from `services/agent_service.py` - Agents NEVER call repositories directly — go through services
LangGraph-specific architecture context. The AI needs to know graph structure before touching any agent code, otherwise it will break state contracts.
## LangGraph architecture Graphs live in: `backend/app/agents/graphs/` Nodes live in: `backend/app/agents/nodes/` State schemas in: `backend/app/agents/state.py` ### Active graphs - `task_planner_graph` — breaks user request → subtasks - `research_graph` — web search + summarization - `code_review_graph` — static analysis + suggestions ### State contract rule - State is a TypedDict defined in state.py - Nodes ONLY receive and return state slices — no side effects - Side effects (DB writes, API calls) happen in services called FROM nodes - Checkpointing: PostgreSQL checkpointer, configured in graph builder ### Node naming convention - Nodes named as verbs: `plan_tasks`, `fetch_context`, `generate_response` - Conditional edges as: `route_by_intent`, `check_completion`

tasks.md
current sprint · active work · what's in progress
This is your most-updated file. It sets the AI's frame of reference for what you're building RIGHT NOW. Without it, the AI assumes all code is greenfield.
Purpose
Tasks.md tells the AI what problem you're currently solving, what exists vs what needs to be built, and what to avoid touching. This prevents scope creep — the AI won't refactor unrelated code while helping you add a feature.
What to include
  • Current sprint goal (1 sentence)
  • Active task with acceptance criteria
  • What's already done (so AI doesn't re-implement)
  • Blockers and known constraints
  • Files being touched this sprint
Template
## Current sprint: AI task planner Goal: user submits a goal → agent breaks it into subtasks → saves to DB ## Active task Building: task_planner_graph in agents/graphs/task_planner.py Acceptance criteria: - Graph accepts: { user_id: str, goal: str } - Returns: { tasks: list[TaskCreate] } - Tasks saved via task_service.create_batch() - Streaming response to frontend via SSE ## Already built - TaskCreate Pydantic schema ✓ - task_service.create_batch() ✓ - SSE endpoint skeleton at POST /tasks/plan ✓ - Frontend SSE hook at frontend/src/hooks/useTaskPlan.ts ✓ ## In progress - [ ] LangGraph graph definition - [ ] plan_tasks node — calls GPT-4o with structured output - [ ] route_by_complexity conditional edge - [ ] Integration test ## Do NOT touch - research_graph — stable, has tests, unrelated - Any Alembic migration files - frontend/src/features/auth — separate PR
Maintenance tip: update tasks.md at the start of each session. Takes 2 minutes, saves 20 minutes of AI context-setting back-and-forth.
conventions.md
naming · patterns · error handling · style rules
This is where you encode your team's taste. Without it, every AI session may produce differently-styled code. With it, the AI writes code that looks like it was written by a single senior engineer.

## Python conventions ### Naming - Files: snake_case.py - Classes: PascalCase - Async functions: prefix with `a_` only for non-obvious async (optional) - Pydantic schemas: suffix with Schema or Request/Response e.g. TaskCreateRequest, TaskResponse, TaskUpdateSchema ### FastAPI patterns - Route handlers: thin — validate, call service, return - Dependencies injected via Depends() — no globals - All DB sessions: AsyncSession via get_async_session dependency - Response models always declared on routes ### Error handling - Raise HTTPException in route layer only - Services raise domain exceptions (TaskNotFoundError, etc.) - Domain exceptions defined in app/core/exceptions.py - Global exception handler in app/core/exception_handlers.py ### Repository pattern - One repo per model: TaskRepository, UserRepository - Repos accept AsyncSession as constructor arg - NO business logic in repos — queries only - Return domain models, not raw rows
## React / TypeScript conventions ### File naming - Components: PascalCase.tsx - Hooks: useCamelCase.ts - Utilities: camelCase.ts - Feature folders: kebab-case/ ### Component patterns - Functional components only — no class components - Props always typed with interface, not type alias - Default exports for page components, named for shared components - No prop drilling beyond 2 levels — use context or query state ### State management - Server state: TanStack Query only — no useState for fetched data - UI state: useState / useReducer — scoped to component - Global UI state: Zustand (store in frontend/src/stores/) - NO Redux, NO MobX ### API calls - Always use generated client from lib/api-client/ - Never use fetch() or axios directly in components - All mutations via useMutation hook with optimistic updates ### Styling - Tailwind utility classes — no inline styles - Component variants via cva() from class-variance-authority - No CSS modules, no styled-components
## LangGraph conventions ### Node contracts - Each node is a pure function: (state: GraphState) -> dict - Return ONLY the keys you're updating - No side effects in nodes — call services for that - Type-annotate state input and return ### State design - State defined as TypedDict in agents/state.py - Use Annotated[list, operator.add] for accumulating fields - Add_messages for conversation history — never replace - Scalar fields overwrite — be explicit in docs ### Error handling in graphs - Use conditional edges to handle error states - Error key in state: errors: list[str] - Terminal error node: `handle_error` — logs, formats response - Never raise exceptions from nodes — catch and add to state.errors ### LLM calls - Always use structured output (with_structured_output) when possible - Model configured in app/core/llm.py — never hardcode model strings - Temperature: 0 for classification/routing, 0.7 for generation
What NOT to put in MD files
the most common mistakes that bloat context and hurt AI accuracy
Bad context is worse than no context. It wastes tokens, confuses the AI, and causes it to reproduce stale patterns.
Content anti-patterns
Never do this
✕ Full code examples in MD
✕ Copy-pasted function signatures
✕ Entire config file contents
✕ Long prose explanations
✕ "Here's how X works" tutorials
✕ TODO lists mixed with docs
✕ Version history / changelogs
✕ Secrets or real env values
Do this instead
✓ File path references
✓ Type signatures only (no body)
✓ Key config variable names
✓ Bullet rules, not essays
✓ Links to relevant docs files
✓ tasks.md for current work
✓ Current state only
✓ Env var names + types only
File-level anti-patterns
  • One giant CLAUDE.md with everything — it loads every time, burns tokens
  • Duplicating info across files — creates contradiction when one gets stale
  • Outdated architecture docs left in place — AI will follow old patterns
  • Nested docs deeper than docs/feature/ — too hard to @mention selectively
  • Auto-generated docs checked into /docs — noise, not signal
The staleness trap
Worst anti-pattern: writing great docs and never updating them. An MD file from 3 months ago with old model names, removed endpoints, or changed patterns will cause the AI to confidently produce broken code. Schedule a docs review every 2 weeks or after every major refactor.

AI session workflow
how to start sessions · what to load · how to prevent context explosion
Session start ritual (30 seconds)
1. Update tasks.md with today's goal
2. Open Cursor / Claude with CLAUDE.md auto-loaded
3. @mention only the docs relevant to today's task
4. Start with: "I'm working on [task from tasks.md]"
Context loading by task type
New API endpoint
Load: @docs/api-schema.md + @docs/conventions.md + @docs/db-schema.md
New React feature
Load: @docs/api-schema.md + @docs/conventions.md
New LangGraph agent
Load: @docs/langgraph.md + @docs/architecture.md + @docs/conventions.md
Bug fix
Load: the specific file with the bug + @docs/conventions.md — avoid loading broad context for targeted fixes
Preventing context explosion
  • Start a new chat session for each distinct task — don't let threads grow past ~20 messages
  • If the AI starts forgetting earlier instructions, re-anchor: paste the relevant section of CLAUDE.md again
  • For long coding tasks, break into steps: design → implement → test — separate sessions
  • Never paste entire files into chat — reference paths, let the AI read the file
  • Use Cursor's @ system to load context selectively, not the entire codebase index
Prompt patterns that work
# Good: anchored, scoped, outcome-clear "Following the conventions in @docs/conventions.md, add a DELETE /tasks/{id} endpoint. It should: - Use the existing TaskRepository.delete() method - Return 404 if task not found (TaskNotFoundError pattern) - Only allow owner or admin (see get_current_user pattern) Do not modify any other files." # Good: referencing existing pattern "Add a useDeleteTask hook following the same pattern as useCreateTask in frontend/src/features/tasks/hooks/useCreateTask.ts" # Bad: vague, no constraints "Add task deletion functionality" # Bad: too broad, context explosion risk "Refactor the entire tasks feature to be better"
