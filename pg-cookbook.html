Chapter 01 · Schema Design

Data Types

Choosing the correct type isn't just academic — it affects storage, query performance, and correctness. Wrong types cause silent bugs that are expensive to fix later.

The Core Type Reference

TypeWhen to UseNotes
BIGINT Primary keys, counters, IDs 8 bytes. Always use for PKs — never regret the headroom. INT maxes at ~2.1B rows.
INTEGER Bounded quantities, foreign counts 4 bytes. Fine when the value will never exceed ~2.1 billion.
NUMERIC(p,s) Money, exact decimals Never FLOAT for currency. Floating point accumulates errors. Use NUMERIC(12,2).
TEXT All variable-length strings No performance penalty over VARCHAR(n). Prefer TEXT + CHECK over VARCHAR(255).
BOOLEAN True/false flags 1 byte. Be careful with NULL — tri-state booleans (T/F/NULL) confuse everything.
TIMESTAMPTZ All timestamps — always Stores in UTC, displays in session TZ. Almost always correct over plain TIMESTAMP.
DATE Calendar dates without time Birthdates, expiry dates, scheduling by day.
UUID Distributed IDs, public-facing IDs Use gen_random_uuid(). Consider UUIDv7 or ULID for time-ordered generation.
JSONB Semi-structured queryable data Binary storage, indexable with GIN. Use for flexible attributes — not as a "schema escape hatch."
ARRAY Small uniform sets (tags, flags) Good for small, primitive lists. Use a junction table for anything relational.
ENUM Fixed value sets Compact and fast, but adding values requires a migration. TEXT + CHECK is more flexible.
INET / CIDR IP addresses Supports subnet operators. Always better than storing as TEXT.
TSVECTOR Full-text search Store pre-computed vectors; pair with GIN indexes. Never store raw text + re-parse on query.

Type Anti-Patterns

✗ Avoid
-- Floats for money (silent precision loss)
price         FLOAT

-- Magic-length VARCHAR
name          VARCHAR(255)

-- Timestamp without timezone
created_at    TIMESTAMP

-- Status as unconstrained string
status        VARCHAR(50)

-- Storing IPs as text
ip_address    TEXT

-- SERIAL (old, deprecated style)
id            SERIAL PRIMARY KEY
✓ Prefer
-- Exact decimal — no surprises
price         NUMERIC(12,2)

-- Unbounded; validate in app layer
name          TEXT

-- Always timezone-aware
created_at    TIMESTAMPTZ DEFAULT NOW()

-- Constrained set with CHECK
status        TEXT CHECK (status IN ('active','paused'))

-- Native IP type with operators
ip_address    INET

-- Modern identity column
id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
💡
Rule of thumb: Store money in cents (INTEGER or BIGINT) or use NUMERIC(12,2). Never FLOAT or DOUBLE PRECISION for anything involving currency.
Chapter 02 · Schema Design

Table Design

A well-structured table is self-documenting, enforces its own integrity, and stays maintainable as your app evolves. These patterns apply to virtually every table you'll create.

Anatomy of a Well-Formed Table

SQL · orders table example
CREATE TABLE orders (

  -- ① Surrogate primary key — always BIGINT IDENTITY, never SERIAL
  id              BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,

  -- ② Foreign keys — always explicit ON DELETE behavior
  user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  store_id        BIGINT NOT NULL REFERENCES stores(id) ON DELETE RESTRICT,

  -- ③ Constrained status — prevents invalid state at DB level
  status          TEXT NOT NULL
                    CHECK (status IN ('pending','paid','shipped','cancelled'))
                    DEFAULT 'pending',

  -- ④ Money in cents — zero floating point ambiguity
  total_cents     BIGINT NOT NULL CHECK (total_cents >= 0),
  currency        TEXT NOT NULL DEFAULT 'USD',

  -- ⑤ Business data
  notes           TEXT,
  metadata        JSONB DEFAULT '{}',

  -- ⑥ Audit columns — on EVERY table, without exception
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ  -- NULL = active (soft delete)
);

-- Index FKs so JOINs are fast
CREATE INDEX ON orders (user_id);
CREATE INDEX ON orders (store_id);
CREATE INDEX ON orders (status) WHERE deleted_at IS NULL;

-- Auto-update updated_at via trigger
CREATE TRIGGER orders_updated_at
  BEFORE UPDATE ON orders
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

The 6 Rules of Table Design

①
Always use a surrogate primary key
Use BIGINT GENERATED ALWAYS AS IDENTITY. Never composite PKs on business columns — they change. Never SERIAL (deprecated). Use UUID when you need globally unique or public-facing IDs.
②
Every column should be NOT NULL unless absence is meaningful
NULLs in the wrong places cause subtle query bugs. Make columns NOT NULL and use sentinel values or separate tables for "optional" relationships.
③
Include created_at, updated_at on every table
Always TIMESTAMPTZ, always DEFAULT NOW(). updated_at should be maintained by a trigger. You will always need these for debugging and auditing — without them you're flying blind.
④
Explicit ON DELETE on every foreign key
Never rely on the default (RESTRICT). Choose deliberately: RESTRICT (safe default), CASCADE (delete children), SET NULL (orphan gracefully), SET DEFAULT. The wrong behavior silently corrupts data.
⑤
Constrain at the database layer, not just the application
Apps have bugs. Migrations bypass the app. Multiple services write to the same DB. Only database constraints are guaranteed to run on every write.
⑥
Use soft deletes thoughtfully
A deleted_at TIMESTAMPTZ column enables recovery and auditing. But always add a partial index WHERE deleted_at IS NULL to every query index, or all queries scan deleted rows.

The updated_at Trigger

SQL · reusable trigger function
-- Create this ONCE, reuse it on every table
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach to any table with:
CREATE TRIGGER <table>_updated_at
  BEFORE UPDATE ON <table>
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
Chapter 03 · Schema Design

Constraints

Constraints are your last line of defense against bad data. Apps have bugs. Scripts bypass the ORM. Only the database runs constraints on every single write, no matter where it comes from.

Constraint Types

NOT NULL
Column must always have a value. The most commonly under-used constraint. Default to NOT NULL unless absence is explicitly meaningful.
UNIQUE
One or more columns must be unique across rows. Creates an implicit B-tree index. Can be partial: UNIQUE WHERE deleted_at IS NULL.
CHECK
Any boolean expression. Use for enums, ranges, email format, positive values, cross-column rules. Far more flexible than it looks.
FOREIGN KEY
Ensures referenced row exists. Always specify ON DELETE and ON UPDATE behavior explicitly. Never omit these.
PRIMARY KEY
NOT NULL + UNIQUE on the identifier column(s). Always use a surrogate bigint identity or UUID.
EXCLUSION
Advanced: no two rows can satisfy a condition together. Classic use: no overlapping date ranges for the same resource.

CHECK Constraint Patterns

SQL · CHECK constraint examples
-- Enum-like constraint (flexible, easy to extend)
status TEXT NOT NULL CHECK (status IN ('draft', 'published', 'archived'))

-- Non-negative amount
quantity INTEGER NOT NULL CHECK (quantity > 0)

-- Valid percentage range
discount_pct NUMERIC(5,2) CHECK (discount_pct BETWEEN 0 AND 100)

-- Cross-column constraint: end must be after start
CONSTRAINT valid_dates CHECK (ends_at > starts_at)

-- Basic email format check
email TEXT NOT NULL CHECK (email ~* '^[^@]+@[^@]+\.[^@]+$')

-- Conditional: discount only if published
CONSTRAINT discount_only_when_published
  CHECK (discount_pct IS NULL OR status = 'published')

-- Add constraint to existing table (VALID = checks new rows only,
-- use NOT VALID + VALIDATE CONSTRAINT for large tables)
ALTER TABLE products
  ADD CONSTRAINT positive_price
  CHECK (price_cents > 0) NOT VALID;

ALTER TABLE products
  VALIDATE CONSTRAINT positive_price;

ON DELETE Behaviors

BehaviorWhat HappensWhen to Use
RESTRICT Error if referenced row has children Safe default. Prevents accidental orphan creation. Use when deletion must be explicit.
CASCADE Delete children when parent deleted Dependent data (order → order_items). Be careful — cascades can propagate deeper than expected.
SET NULL FK column set to NULL on parent delete Optional relationships. Column must allow NULL. Useful for soft references.
SET DEFAULT FK set to column's DEFAULT value Rare. Useful for "unknown" or placeholder parent rows.
NO ACTION Like RESTRICT but deferred Used with DEFERRABLE constraints for within-transaction reordering.
⚠️
Adding constraints to large tables: Use NOT VALID to skip existing rows (no lock on full scan), then run VALIDATE CONSTRAINT separately which only takes a ShareUpdateExclusiveLock. Never add a plain CHECK to a big table in a single migration.
Chapter 04 · Schema Design

Indexes

Indexes are the single biggest lever for query performance. But every index has a write cost. Here's how to think about when to add them, which type to use, and what to avoid.

Index Types

TypeBest ForExample
B-Tree Default. Equality, range queries, ORDER BY, LIKE 'foo%' id, status, created_at, email
GIN JSONB containment (@>), arrays, full-text search (@@) metadata JSONB, tags TEXT[], tsvector columns
GiST Geometric data, ranges (&&, @>, <@), PostGIS daterange, tsrange, geometry columns
HASH Equality-only lookups on very large values Rarely needed; B-Tree is usually better
BRIN Very large tables with natural physical ordering (time-series, IoT) sensor_readings.recorded_at on append-only tables

Index Patterns

SQL · index examples
-- ① Always index foreign keys
CREATE INDEX idx_orders_user_id ON orders (user_id);

-- ② Partial index — only index the rows you query
--   (far smaller and faster than indexing all rows)
CREATE INDEX idx_orders_pending
  ON orders (created_at)
  WHERE status = 'pending';

-- ③ Composite index — column order matters!
--   Covers: (user_id), (user_id, status), but NOT (status) alone
CREATE INDEX idx_orders_user_status
  ON orders (user_id, status);

-- ④ Covering index (INCLUDE) — index-only scan, no heap fetch
CREATE INDEX idx_orders_user_covering
  ON orders (user_id)
  INCLUDE (status, total_cents, created_at);

-- ⑤ Expression index — index the result of a function
CREATE INDEX idx_users_email_lower
  ON users (lower(email));
-- Query must use the same expression: WHERE lower(email) = ...

-- ⑥ GIN for JSONB queries
CREATE INDEX idx_products_metadata
  ON products USING GIN (metadata);

-- ⑦ CONCURRENTLY — never blocks reads/writes (use in migrations!)
CREATE INDEX CONCURRENTLY idx_events_user_id
  ON events (user_id);
-- Cannot be run inside a transaction block

Composite Index Column Order

Put the most selective, equality-tested column first. Range-tested columns go last.

✗ Wrong order
-- Query: WHERE user_id = 5 AND status = 'paid'
-- This index won't help for user_id alone

CREATE INDEX ON orders (status, user_id);

-- status has low cardinality (few distinct values)
-- Postgres may skip this index entirely
✓ Right order
-- High-cardinality (user_id) goes first
-- Also works for queries on user_id alone

CREATE INDEX ON orders (user_id, status);

-- Rule: equality columns first,
-- range columns (created_at) last
🚫
Never add indexes blindly. Each index adds ~20-30% write overhead. Over-indexing tables with heavy INSERT/UPDATE/DELETE is worse than no index. Profile with EXPLAIN ANALYZE first, then add the minimal index that fixes it.

Find Slow Queries & Unused Indexes

SQL · diagnostics
-- Indexes never used (safe to drop candidates)
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;

-- Tables missing indexes on FK columns (sequential scans)
SELECT relname, seq_scan, seq_tup_read, idx_scan
FROM pg_stat_user_tables
WHERE seq_scan > 50
ORDER BY seq_scan DESC;

-- Bloated indexes (fragmented, needs REINDEX)
SELECT indexrelname,
       pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC
LIMIT 10;
Chapter 05 · Schema Design

Relationships

How you model relationships between entities is the core of relational database design. Get these patterns right and everything else becomes easier.

One-to-Many (Most Common)

SQL
-- One user → many posts
CREATE TABLE posts (
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id    BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title      TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index the FK — always
CREATE INDEX ON posts (user_id);

Many-to-Many (Junction Table)

SQL
-- Many posts ↔ many tags
CREATE TABLE post_tags (
  post_id    BIGINT NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  tag_id     BIGINT NOT NULL REFERENCES tags(id)  ON DELETE CASCADE,

  -- Composite PK prevents duplicates
  PRIMARY KEY (post_id, tag_id),

  -- Junction tables can have their own data
  added_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  added_by   BIGINT REFERENCES users(id)
);

-- Index the reverse direction for tag → posts lookups
CREATE INDEX ON post_tags (tag_id);

One-to-One

SQL
-- User profile is 1-to-1 with user
-- Use UNIQUE on the FK to enforce the "one" side
CREATE TABLE user_profiles (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id     BIGINT NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,
  bio         TEXT,
  avatar_url  TEXT
);

Self-Referential (Hierarchies)

SQL
-- Categories with parent categories (tree structure)
CREATE TABLE categories (
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  parent_id  BIGINT REFERENCES categories(id) ON DELETE RESTRICT,
  name       TEXT NOT NULL
);

CREATE INDEX ON categories (parent_id);

-- Query full hierarchy with recursive CTE
WITH RECURSIVE tree AS (
  SELECT * FROM categories WHERE id = 1
  UNION ALL
  SELECT c.* FROM categories c
  JOIN tree t ON c.parent_id = t.id
)
SELECT * FROM tree;
💡
Always index FK columns. PostgreSQL does NOT automatically create indexes on foreign key columns (only on the PK side). An un-indexed FK causes a full sequential scan every time you JOIN or do a cascade check.
Chapter 06 · Schema Design

Normalization

Normalization is the process of organizing data to reduce redundancy and prevent update anomalies. Here's a practical, pattern-based understanding — not just theory.

Why Normalize?

No Data Duplication
Each piece of information lives in exactly one place. Update it once, consistent everywhere.
No Update Anomalies
Changing a customer's name doesn't require updating 50 order rows — just one users row.
Referential Integrity
FK constraints only work when related data is in separate tables. Embedded data can't be constrained.

First Normal Form (1NF) — Atomic Values

Every column stores one value. No comma-separated lists, no repeating groups.

✗ Violates 1NF
-- Multiple phones in one column
users
┌─────┬──────────────────────────┐
│ id  │ phone_numbers            │
├─────┼──────────────────────────┤
│  1  │ '555-1234, 555-5678'    │
│  2  │ '555-9999'              │
└─────┴──────────────────────────┘

-- Can't query individual phone numbers
-- Can't index them
-- Impossible to FK-constrain
✓ 1NF Compliant
-- Each phone in its own row
user_phones
┌─────┬─────────┬────────────┐
│ id  │ user_id │ phone      │
├─────┼─────────┼────────────┤
│  1  │    1    │ 555-1234   │
│  2  │    1    │ 555-5678   │
│  3  │    2    │ 555-9999   │
└─────┴─────────┴────────────┘

-- Queryable, indexable, constrained

Second Normal Form (2NF) — No Partial Dependencies

Every non-key column must depend on the whole primary key. Applies when you have composite PKs.

✗ Violates 2NF
-- PK is (order_id, product_id)
-- product_name depends only on product_id
-- → updating a product name requires
--   updating every order row

order_items
┌──────────┬────────────┬──────────────┬───────┐
│ order_id │ product_id │ product_name │  qty  │
├──────────┼────────────┼──────────────┼───────┤
│    1     │    42      │   Widget X   │   3   │
│    2     │    42      │   Widget X   │   1   │
└──────────┴────────────┴──────────────┴───────┘
✓ 2NF Compliant
-- product_name lives in products table
-- order_items only has order-specific data

products            order_items
┌─────┬──────────┐  ┌──────────┬────────────┬─────┐
│ id  │ name     │  │ order_id │ product_id │ qty │
├─────┼──────────┤  ├──────────┼────────────┼─────┤
│ 42  │ Widget X │  │    1     │     42     │  3  │
└─────┴──────────┘  │    2     │     42     │  1  │
                    └──────────┴────────────┴─────┘

Third Normal Form (3NF) — No Transitive Dependencies

Non-key columns must depend only on the PK, not on other non-key columns.

✗ Violates 3NF
-- zip_code → city → state (transitive chain)
-- Changing a zip code's city means
-- scanning the entire users table

users
┌─────┬──────┬──────────┬─────────┐
│ id  │ name │ zip_code │  city   │
├─────┼──────┼──────────┼─────────┤
│  1  │ Ali  │  10001   │ New York│
│  2  │ Sara │  10001   │ New York│
└─────┴──────┴──────────┴─────────┘
✓ 3NF Compliant
-- city is a property of zip_code, not user

zip_codes           users
┌──────────┬──────────┐  ┌─────┬──────┬──────────┐
│ zip_code │ city     │  │ id  │ name │ zip_code │
├──────────┼──────────┤  ├─────┼──────┼──────────┤
│  10001   │ New York │  │  1  │ Ali  │  10001   │
└──────────┴──────────┘  │  2  │ Sara │  10001   │
                         └─────┴──────┴──────────┘
💡
When to denormalize: For read-heavy analytics or reporting, controlled denormalization (storing pre-computed values) is a valid trade-off. Use materialized views or separate reporting tables — not your core transactional schema.
Chapter 07 · Migrations

Migration Basics

Migrations are versioned, reproducible scripts that evolve your database schema alongside your code. Never change a database schema manually in production.

What Is a Migration?

A migration is a file that contains SQL (or ORM code) to move the database from state A to state B. It always has two operations: up (apply the change) and down (roll it back).

SQL · migration file example
-- migrations/20240315120000_add_phone_to_users.sql

-- ── UP ─────────────────────────────────────────
ALTER TABLE users
  ADD COLUMN phone TEXT,
  ADD COLUMN phone_verified_at TIMESTAMPTZ;

CREATE INDEX CONCURRENTLY idx_users_phone ON users(phone)
  WHERE phone IS NOT NULL;

-- ── DOWN ───────────────────────────────────────
-- (in separate down file or section)
DROP INDEX CONCURRENTLY IF EXISTS idx_users_phone;
ALTER TABLE users
  DROP COLUMN IF EXISTS phone,
  DROP COLUMN IF EXISTS phone_verified_at;

Migration File Naming

Use a timestamp prefix so files sort chronologically and avoid conflicts in team environments:

20240315120000_add_phone_to_users.sql
20240316083000_create_notifications_table.sql
20240320140000_add_index_orders_status.sql

Format: YYYYMMDDHHMMSS_descriptive_snake_case_name

Migration Tooling

ToolLanguageBest For
FlywayAny (SQL files)Simple, reliable, SQL-first. Great for any backend language.
LiquibaseAny (XML/YAML/SQL)Enterprise features, complex change tracking, multi-DB.
golang-migrateGoLightweight Go projects. Pure SQL files.
AlembicPythonSQLAlchemy projects. Supports auto-generation from models.
Prisma MigrateNode.jsTypeScript/Node projects. Auto-generates from schema file.
ActiveRecordRubyRails apps. Ruby DSL + rollback support built-in.
Knex / UmzugNode.jsNode apps without a full ORM. SQL or JS-based migrations.

The Golden Rules of Migrations

✗
Never edit a migration that has run in any environment
Create a new migration to fix or revert a previous one. Other developers and environments have already run the original — editing it creates inconsistency.
✗
Never make schema changes directly in production
No psql sessions, no manual ALTERs "just this once." Schema must flow through version control → migration tool → production. Always.
✓
One migration = one logical change
Keep migrations focused. "Add phone column" and "add notification table" should be separate files. Easier to review, easier to roll back individually.
✓
Test migrations on a production-sized data copy
A migration that takes 10ms on a dev database with 100 rows may take 45 minutes and lock the table on 50 million rows in production.
Chapter 08 · Migrations

Safe Migration Patterns

These patterns let you evolve your schema without breaking running application code or causing downtime.

The Expand/Contract Pattern

The safest way to rename columns, change types, or restructure tables. Split into phases deployed separately.

Phase 1 — Expand
Add the new column alongside the old one
Add the new column (nullable). Both old and new app code still work because nothing is removed yet.
Phase 2 — Backfill
Populate the new column from the old one
Run a backfill query in batches. New writes should write to both columns (dual-write in app code or via trigger).
Phase 3 — Switch
Deploy app code reading from the new column
Update app to read from the new column. Add NOT NULL constraint to new column. Monitor for errors.
Phase 4 — Contract
Drop the old column
Once all app code is off the old column, drop it in a separate deployment. Safe because no code references it anymore.

Renaming a Column Safely

SQL + pseudocode
-- GOAL: rename users.name → users.full_name

-- Migration 1: Add new column (nullable)
ALTER TABLE users ADD COLUMN full_name TEXT;

-- Migration 2: Backfill in batches
UPDATE users SET full_name = name
WHERE full_name IS NULL
  AND id BETWEEN 1 AND 10000; -- repeat in chunks

-- App code: dual-write to both columns during transition

-- Migration 3: Add NOT NULL after backfill is complete
ALTER TABLE users
  ALTER COLUMN full_name SET NOT NULL;

-- Migration 4: Drop old column (after app reads only full_name)
ALTER TABLE users DROP COLUMN name;

Adding NOT NULL to an Existing Column

SQL · safe approach for large tables
-- WRONG: locks entire table while scanning all rows
ALTER TABLE orders ALTER COLUMN status SET NOT NULL; -- ✗

-- RIGHT: use a CHECK constraint with NOT VALID first

-- Step 1: Add constraint without scanning existing rows (instant)
ALTER TABLE orders
  ADD CONSTRAINT orders_status_not_null
  CHECK (status IS NOT NULL) NOT VALID;

-- Step 2: Backfill nulls if any exist
UPDATE orders SET status = 'pending' WHERE status IS NULL;

-- Step 3: Validate constraint (ShareUpdateExclusiveLock — does NOT block reads/writes)
ALTER TABLE orders VALIDATE CONSTRAINT orders_status_not_null;

-- Step 4: Postgres 12+ recognizes validated CHECK → can now use ALTER NOT NULL cheaply
ALTER TABLE orders ALTER COLUMN status SET NOT NULL;

Backfill Strategy for Large Tables

SQL · batch backfill
-- Never UPDATE the entire table at once — it locks rows for too long
-- Batch by PK range, with a small sleep between batches

DO $$
DECLARE
  batch_size  INT := 5000;
  last_id     BIGINT := 0;
  max_id      BIGINT;
BEGIN
  SELECT MAX(id) INTO max_id FROM orders;

  WHILE last_id < max_id LOOP
    UPDATE orders
    SET full_status = status
    WHERE id > last_id
      AND id <= last_id + batch_size
      AND full_status IS NULL;

    last_id := last_id + batch_size;
    PERFORM pg_sleep(0.05); -- 50ms pause between batches
  END LOOP;
END; $$;
Chapter 09 · Migrations

Danger Zones

These operations can lock your tables, rewrite all rows, or take down production. Know them. Fear them appropriately.

Locking Levels

Every DDL statement acquires a lock. Most destructive operations take an AccessExclusiveLock — this blocks ALL reads and writes until the operation completes.

Lock LevelBlocksTypical Operations
AccessExclusiveLock Everything — reads, writes, vacuums DROP TABLE, TRUNCATE, most ALTER TABLE operations
ShareRowExclusiveLock Writes, other DDL CREATE TRIGGER, some FK operations
ShareUpdateExclusiveLock Other DDL only, NOT reads/writes CREATE INDEX CONCURRENTLY, VALIDATE CONSTRAINT, VACUUM
ShareLock Writes only CREATE INDEX (non-concurrent)
🔥
The queue problem: Even if your ALTER TABLE finishes in 2 seconds, if there's a long-running query when it starts, the ALTER waits behind it. And every new query waits behind the ALTER. A 30-second idle transaction can cause a 5-minute production outage.

Operations That Rewrite the Entire Table

These operations copy every row to a new heap, holding an AccessExclusiveLock the entire time. On a 100GB table, that could be hours.

!
ALTER COLUMN … TYPE (type change)
Changing INTEGER → BIGINT, VARCHAR(50) → TEXT, etc. Full table rewrite. Use the expand/contract pattern instead.
!
ADD COLUMN with a volatile DEFAULT
In PostgreSQL < 11: adding a column with DEFAULT rewrites every row. PostgreSQL 11+ handles constant defaults without rewrite — but avoid DEFAULT NOW() (volatile) on large tables.
!
CREATE INDEX (without CONCURRENTLY)
Blocks writes for the entire index build. Always use CREATE INDEX CONCURRENTLY. Cannot run inside a transaction block.
!
ALTER TABLE … SET NOT NULL (pre-PG12 or unvalidated)
Full table scan + lock to verify no NULLs exist. Use the NOT VALID → VALIDATE CONSTRAINT pattern instead (see Chapter 08).
!
Adding a FOREIGN KEY without NOT VALID
Scans the entire child table to validate every row. Use NOT VALID first, then VALIDATE CONSTRAINT separately.

Safe vs Dangerous Side-by-Side

✗ Dangerous
-- Locks table, rewrites rows (type change)
ALTER TABLE users
  ALTER COLUMN age TYPE BIGINT;

-- Blocks all writes during build
CREATE INDEX idx_orders_user_id
  ON orders (user_id);

-- Full scan + lock on large table
ALTER TABLE orders
  ADD CONSTRAINT fk_user
  FOREIGN KEY (user_id) REFERENCES users(id);

Chapter 10 · Migrations

Zero-Downtime Migrations

Ship schema changes to production while handling live traffic. This requires coordination between database changes and application deployments.

The Core Principle

✅
Database changes must be backward-compatible with both the old and new version of your app code. Because during a rolling deploy, both versions are running simultaneously against the same database.

What Is and Isn't Safe in a Live Deploy

OperationSafe?Reason
ADD COLUMN (nullable, no default) ✓ Safe Old code ignores new column. No rewrite. Instant in PG 11+.
ADD COLUMN with constant DEFAULT ✓ Safe (PG11+) PG11+ stores the default in catalog without rewriting rows.
ADD COLUMN with volatile DEFAULT (NOW()) ⚠ Careful Forces a table rewrite in all versions. Use nullable first, backfill separately.
CREATE INDEX CONCURRENTLY ✓ Safe Builds in background. No write locks. Cannot be inside a transaction.
DROP COLUMN ✗ Only after deploy Old code will error if it references the dropped column. Deploy app change first.
RENAME COLUMN ✗ Never directly Old code breaks immediately. Use expand/contract pattern across multiple deploys.
ALTER COLUMN TYPE ✗ Never directly Table rewrite + type change can break existing app queries. Use expand/contract.
DROP TABLE ✗ Only after deploy Must ensure zero code references the table before dropping.
ADD NOT NULL constraint ⚠ Use NOT VALID Use the CHECK NOT NULL / VALIDATE pattern from Chapter 08.

Full Example: Add a Required Column

1
Migration: Add nullable column
Deploy this before any app code. Old app works fine — ignores the new column.
2
App deploy: Write to both old + new column
Dual-write. New rows populate the new column. Old rows are still null.
3
Migration: Backfill existing rows
Batch-update existing rows in chunks. After this, all rows have the new value.
4
Migration: Add NOT NULL constraint (NOT VALID → VALIDATE)
Enforce the constraint without a full table lock.
5
App deploy: Read from new column, stop dual-write
App is now fully on the new column. Old column no longer written.
6
Migration: Drop old column
Safe to drop now — no code references it. The final cleanup step.
💡
Use lock_timeout in every migration: Add SET lock_timeout = '3s'; at the top of each migration script. If the migration can't acquire a lock within 3 seconds, it aborts instead of queueing all subsequent queries behind it.

Schema Operations

SQL
-- Add column (safe, nullable first)
ALTER TABLE t ADD COLUMN col TEXT;
ALTER TABLE t ADD COLUMN col TEXT NOT NULL DEFAULT ''; -- PG11+

-- Drop column
ALTER TABLE t DROP COLUMN IF EXISTS col;

-- Rename column
ALTER TABLE t RENAME COLUMN old_name TO new_name;

ALTER TABLE old_name RENAME TO new_name;

-- Change type (CAREFUL — may rewrite table)
ALTER TABLE t ALTER COLUMN col TYPE BIGINT

-- Set default

ALTER TABLE t ALTER COLUMN col DROP DEFAULT;

-- Set / drop NOT NULL
ALTER TABLE t ALTER COLUMN col SET NOT NULL;
ALTER TABLE t ALTER COLUMN col DROP NOT NULL;

Constraints

SQL
-- Add named constraint
ALTER TABLE t ADD CONSTRAINT name CHECK (col > 0);
ALTER TABLE t ADD CONSTRAINT name UNIQUE (col);
ALTER TABLE t ADD CONSTRAINT name
  FOREIGN KEY (col) REFERENCES other(id) NOT VALID;

-- Validate deferred constraint
ALTER TABLE t VALIDATE CONSTRAINT name;

-- Drop constraint
ALTER TABLE t DROP CONSTRAINT IF EXISTS name;

-- Partial unique constraint (unique among active rows)
CREATE UNIQUE INDEX idx_t_col_active
  ON t (col) WHERE deleted_at IS NULL;

Indexes

SQL
-- Standard B-tree index
CREATE INDEX idx_t_col ON t (col);

-- Non-blocking (CONCURRENTLY)
CREATE INDEX CONCURRENTLY idx_t_col ON t (col);

-- Partial index

REINDEX INDEX CONCURRENTLY idx_t_col;

       query, state

FROM pg_constraint c
JOIN pg_attribute a ON a.attnum = c.conkey[1] AND a.attrelid = c.conrelid

  AND NOT EXISTS (

    WHERE i.indrelid = c.conrelid AND c.conkey[1] = ANY(i.indkey)

Pre-Deploy Checklist

Run through every item before pushing a migration to production. This checklist has prevented countless production incidents.

Before Writing the Migration

✓
Can this be split into smaller migrations?
Prefer multiple small, focused migrations over one large combined one. Easier to test, easier to roll back.
✓
Is the database change backward-compatible with the current app?
The migration may run while the old app version is still live. Adding columns = safe. Dropping columns = needs app deploy first.
⚠
Does any step involve a table with >1M rows?
Flag immediately. Every operation needs to use CONCURRENTLY or NOT VALID patterns. Test on a copy of production data.

Writing the Migration

✓
lock_timeout and statement_timeout are set at the top
SET lock_timeout = '3s'; SET statement_timeout = '120s'; — never let a migration queue the entire database.
✓
All new indexes use CONCURRENTLY
Every CREATE INDEX in a migration uses CREATE INDEX CONCURRENTLY. No exceptions on any table with live traffic.
✓
New FK constraints use NOT VALID → VALIDATE pattern
Adding a FK with NOT VALID is instant. VALIDATE CONSTRAINT only needs ShareUpdateExclusiveLock — won't block queries.
✓
New non-nullable columns have a DEFAULT or are nullable
Never add a NOT NULL column without a DEFAULT to an existing table. New rows will fail until the app is deployed.
✓
A down/rollback migration exists and is tested
Can you roll back this migration in under 30 seconds? If not, your rollback strategy is "fix it forward" — document that explicitly.

Before Running on Production

✓
Tested on a production data snapshot
Ran the migration against a recent production dump. You know the actual runtime, not an estimate from a dev database.
✓
Checked for long-running transactions before running
Run SELECT pid, now() - query_start AS age, query FROM pg_stat_activity WHERE state != 'idle' ORDER BY age DESC; and wait for or kill any long blockers.
✓
EXPLAIN ANALYZE run on affected queries
Verified that queries using the new schema/index are using the expected query plan. No accidental seq scans.
⚠
Someone is on-call to monitor during the migration
Never run a production migration unattended. Have error logs, application metrics, and database monitoring visible. Have rollback steps ready to execute.
⚠
Off-peak window chosen for risky migrations
Even "safe" migrations can have unexpected edge cases. Schedule large migrations during low-traffic periods. Never on a Friday.
🚫
Never run migrations on a Friday afternoon. If something goes wrong, you want your full team available to respond — not heading into the weekend.
